## 一、电商交易链路

> 秒杀是电商的一个点，把整条交易链路串起来，面试最高频。

### 1.1 购物车 → 下单 → 支付 → 履约

| 环节 | 核心难点 | 技术要点 |
|------|---------|---------|
| 购物车 | 未登录合并、库存预占 | Redis Hash 存储、登录后合并策略 |
| 下单 | 幂等性、库存扣减 | 分布式锁 + Lua 原子操作（已有） |
| 支付 | 回调幂等、超时关闭 | RocketMQ 延迟消息（已有思路） |
| 履约 | 状态流转、异常补偿 | DDD 状态机（已有聚合根经验） |
| 退款 | 部分退款、退款回调 | 事务消息 + 对账 |

**支付回调幂等设计**：
```
支付平台回调 → 验签 → 查询订单状态
    ├── 已支付 → 直接返回成功（幂等）
    ├── 待支付 → 更新状态 + 通知下游 → 返回成功
    └── 已关闭 → 记录异常 → 返回成功（不重试）
```

### 1.2 优惠券系统

| 模块 | 核心难点 | 技术要点 |
|------|---------|---------|
| 券模板 | 满减/折扣/随机金额 | 策略模式（已有） |
| 领券 | 限量抢领、防重领 | Redis DECR + SISMEMBER |
| 核销 | 叠加规则、优先级计算 | 规则引擎、责任链（已有） |
| 过期 | 批量过期处理 | 延迟队列（已有思路） |

**优惠叠加规则**：互斥券 → 平台券 → 店铺券 → 满减（责任链模式）

### 1.3 库存系统

| 场景 | 核心难点 | 技术要点 |
|------|---------|---------|
| 预扣库存 | 下单锁定、支付确认 | Redis Lua 预扣 + DB 最终扣减 |
| 库存回补 | 支付超时/取消回补 | 延迟队列（已有） |
| 分库分表 | 百万级 SKU | ShardingSphere-JDBC |
| 库存对账 | 缓存与 DB 一致性 | Canal 监听 Binlog + 定时对账 |

## 二、即时通讯（IM）

> 面试第二大高频场景，Netty 是 Java 网络编程天花板。

### 2.1 核心架构

```
客户端 A → WebSocket → IM Gateway (Netty)
                          ↓
                     消息路由（Redis 查询在线状态）
                          ↓
                ┌─────────┼─────────┐
                ↓         ↓         ↓
           在线推送    离线存储    群聊广播
           (直接推送)  (DB+MQ)   (异步扇出)
```

### 2.2 核心模块

| 模块 | 核心难点 | 技术要点 |
|------|---------|---------|
| 长连接 | 百万连接、心跳检测 | Netty、WebSocket、心跳机制 |
| 消息投递 | 不丢、不重、有序 | ACK 机制、消息重试、雪花 ID |
| 离线消息 | 离线期间消息存储 | 消息表 + 拉取同步 |
| 群聊 | 大群扩散写、已读回执 | 读扩散 vs 写扩散、Redis Bitmap |
| 消息搜索 | 聊天记录全文检索 | Elasticsearch |

### 2.3 消息可靠投递

```
发送方 → 服务端（写 DB + ACK 发送方）→ 推送接收方
                                            ↓
                                    接收方 ACK → 标记已送达
                                            ↓
                                    超时未 ACK → 重试推送（最多 3 次）
```

### 2.4 新增学习点

- **Netty**：线程模型、编解码器、零拷贝、内存池
- **WebSocket**：STOMP 协议、连接管理
- **消息存储**：消息表分表策略、冷热分离

## 三、内容平台 / Feed 流

> ES 搜索 + Feed 流设计是中高级面试必问。

### 3.1 Feed 流架构

| 模式 | 原理 | 优点 | 缺点 | 适用 |
|------|------|------|------|------|
| **推模式** | 发布时写入所有粉丝收件箱 | 读快 | 大 V 写放大 | 粉丝少 |
| **拉模式** | 读取时实时拉取关注人动态 | 写快 | 读慢 | 粉丝多 |
| **推拉结合** | 普通用户推、大 V 拉 | 平衡 | 复杂 | **推荐** |

**推拉结合核心**：
```
发布内容 → 判断粉丝数
    ├── < 10000 → 推模式（写入粉丝收件箱）
    └── >= 10000 → 大 V 标记（读时拉取）

读取 Feed → 合并（收件箱 + 大 V 拉取）→ 排序 → 返回
```

### 3.2 核心模块

| 模块 | 核心难点 | 技术要点 |
|------|---------|---------|
| 发布 | 内容存储、媒体上传 | OSS 对象存储、异步审核 |
| Feed 流 | 推拉结合、分页游标 | Redis List/ZSet、时间线 |
| 点赞/收藏 | 高并发计数 | Redis INCR + 异步落库 |
| 评论 | 楼中楼、@提醒 | 树形结构、事件通知 |
| 搜索 | 标题/内容/标签检索 | **Elasticsearch** 分词、聚合 |
| 推荐 | 个性化推荐 | 协同过滤、向量相似度 |

### 3.3 点赞计数方案

```
用户点赞 → Redis SISMEMBER（判重）
    ├── 已赞 → 取消赞（SREM + DECR）
    └── 未赞 → 点赞（SADD + INCR）
                ↓
        异步消息 → DB 落库（批量合并）
```

### 3.4 新增学习点

- **Elasticsearch**：分词器、索引设计、聚合查询、深分页
- **Feed 流**：时间线设计、大 V 问题、分页游标
- **CDN**：静态资源加速、图片处理

## 四、权限管理系统（RBAC/ABAC）

> Spring Security 是 java26s 待学习的 P1，做完权限系统直接掌握。

### 4.1 RBAC 模型

```
User ←── UserRole ──→ Role ←── RolePermission ──→ Permission
用户      多对多        角色       多对多              权限
```

### 4.2 四级权限

| 级别 | 说明 | 实现方式 |
|------|------|---------|
| 菜单权限 | 控制页面可见性 | 前端路由守卫 + 后端接口 |
| 按钮权限 | 控制操作可用性 | `v-hasPerm` 指令 + 后端校验 |
| 接口权限 | 控制 API 访问 | `@PreAuthorize("hasAuthority('user:add')")` |
| 数据权限 | 控制数据可见范围 | MyBatis 拦截器注入 SQL 条件 |

### 4.3 数据权限（行级隔离）

```java
// 数据权限注解
@DataScope(deptAlias = "d", userAlias = "u")

// MyBatis 拦截器自动拼接
SELECT * FROM order o
WHERE o.deleted = 0
  AND o.dept_id IN (SELECT dept_id FROM sys_role_dept WHERE role_id = ?)
```

### 4.4 多租户 SaaS 隔离方案

| 方案 | 隔离级别 | 优点 | 缺点 |
|------|---------|------|------|
| 独立数据库 | 最强 | 完全隔离 | 成本高 |
| 共享数据库独立 Schema | 强 | 较好隔离 | 运维复杂 |
| 共享表 + tenant_id | 弱 | 成本最低 | 需处处加条件 |

### 4.5 单点登录（SSO）

```
用户访问系统 A → 未登录 → 跳转 SSO 认证中心
    → 输入账号密码 → 颁发 Token → 回调系统 A
    → 访问系统 B → SSO 自动登录（Cookie 共享）
```

### 4.6 新增学习点

- **Spring Security**：FilterChain、AuthenticationManager、UserDetailsService
- **JWT**：Access Token + Refresh Token、Token 续期
- **OAuth2**：授权码模式、客户端模式、资源服务器

## 六、外卖/打车（LBS 类）

> Redis GEO 是新知识点，状态机设计可以复用 DDD。

### 6.1 核心架构

```
用户下单 → 智能派单 → 骑手/司机接单 → 实时位置推送 → 送达确认
              ↓
         就近匹配（Redis GEOSEARCH）
         负载均衡（评分模型）
```

### 6.2 核心模块

| 模块 | 核心难点 | 技术要点 |
|------|---------|---------|
| 附近搜索 | 范围查询、排序 | **Redis GEO** GEOSEARCH |
| 实时位置 | 高频更新、推送 | WebSocket + Redis GEO |
| 智能派单 | 就近 + 评分 | 距离算法 + 评分模型 |
| 订单状态 | 复杂状态流转 | DDD 聚合根（已有） |
| 地理围栏 | 区域判断 | GeoHash 精度计算 |

### 6.3 Redis GEO 操作

```bash
GEOADD   drivers 116.40 39.90 driver:1001     # 添加位置
GEOSEARCH drivers FROMLONLAT 116.41 39.91 BYRADIUS 3 km  # 搜索附近
GEODIST  drivers driver:1001 driver:1002 m     # 计算距离
```

### 6.4 智能派单评分模型

```
综合评分 = 距离权重(0.4) × 距离分
         + 评分权重(0.3) × 骑手评分
         + 完成率权重(0.2) × 完成率
         + 空闲权重(0.1) × 空闲时间
```

### 6.5 新增学习点

- **Redis GEO**：GEOADD/GEOSEARCH/GEODIST
- **GeoHash**：编码原理、精度与范围
- **WebSocket**：实时推送

## 七、数据同步 / ETL

> Canal + Redis 缓存一致性是面试高频追问点。

### 7.1 Binlog 监听方案

```
MySQL Binlog → Canal Server → Canal Client
                                    ↓
                        ┌───────────┼───────────┐
                        ↓           ↓           ↓
                   更新 Redis    写入 ES     通知下游
                   (缓存一致)   (搜索索引)   (事件驱动)
```

### 7.2 缓存一致性方案

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 先删缓存再更新 DB | 删除缓存 → 更新 DB | 简单 | 并发不一致 |
| 先更新 DB 再删缓存 | 更新 DB → 删除缓存 | 较好 | 仍有短暂不一致 |
| **延迟双删** | 删缓存 → 更新 DB → 延迟再删 | 较可靠 | 延迟时间难定 |
| **Canal 监听** | 监听 Binlog → 更新缓存 | 最可靠 | 架构复杂 |

### 7.3 数据迁移方案

```
百万级数据迁移：
1. 全量同步（分页查询写入目标库）
2. 增量同步（Canal 监听 Binlog）
3. 数据校验（抽样对比 + 全量校验）
4. 切换流量（灰度 → 全量）
```

### 7.4 新增学习点

- **Canal**：部署、配置、客户端开发
- **Debezium**：CDC（Change Data Capture）
- **数据一致性**：最终一致性保障

## 八、AI 集成应用（2026 趋势）

> 2026 年趋势，Spring AI 是 Spring 生态官方方案。

### 8.1 智能客服架构

```
用户提问 → 意图识别 → 知识库检索（RAG）
                          ↓
                    向量数据库（Milvus）
                    相似文档 Top-K
                          ↓
                    LLM 生成回答 → 返回用户
```

### 8.2 核心模块

| 模块 | 核心难点 | 技术要点 |
|------|---------|---------|
| 知识库 | 文档切分、向量化 | Embedding 模型、分块策略 |
| 向量检索 | 相似度搜索 | Milvus / PgVector |
| 对话管理 | 多轮上下文 | 会话存储、Token 管理 |
| 内容审核 | 敏感词/图片 | LLM 审核 + 规则引擎 |

### 8.3 Spring AI 快速集成

```java
@Autowired
ChatClient chatClient;

public String chat(String message) {
    return chatClient.prompt()
        .user(message)
        .call()
        .content();
}
```

### 8.4 新增学习点

- **Spring AI**：ChatClient、Embedding、VectorStore
- **RAG**：检索增强生成、知识库构建
- **MCP 协议**：AI 工具调用标准

## 十、多租户 SaaS 平台

> SaaS 架构是企业级开发核心能力。

### 10.1 租户隔离方案

```
请求 → 租户识别（Header/子域名）→ 数据源路由
                                        ├── 独立数据库（大客户）
                                        ├── 独立 Schema（中客户）

```

### 10.2 核心模块

| 模块 | 核心难点 | 技术要点 |
|------|---------|---------|
| 租户管理 | 租户注册、初始化 | 自动建库/建表、初始化数据 |
| 订阅计费 | 套餐管理、用量统计 | 状态机 + 定时任务 |
| 功能开关 | 按套餐启停功能 | Feature Flag、注解控制 |
| 数据导出 | 租户数据导出 | 异步导出 + 下载链接 |
| 运维管理 | 租户监控、限流 | Prometheus 多维监控 |

### 10.3 订阅状态机

```
试用中 → 已激活 → 到期 → 已停用
              ↓        ↓
           续费 ←──────┘
              ↓
           升级套餐
```

### 10.4 新增学习点

- **多租户**：AbstractRoutingDataSource、Schema 路由
- **计费系统**：Stripe/支付宝当面付集成
- **Feature Flag**：功能开关设计
