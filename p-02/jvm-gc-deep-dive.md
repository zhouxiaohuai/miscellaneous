## 一、GC 基础概念

### 1.1 什么是垃圾回收

垃圾回收（Garbage Collection，GC）是 JVM 自动管理内存的机制，负责回收不再使用的对象所占用的内存空间。

### 1.2 判断对象存活的方法

#### 引用计数法

```
原理：每个对象维护一个引用计数器
优点：实现简单，判断效率高
缺点：无法解决循环引用问题

示例：
  Object A = new Object();  // A.refCount = 1
  Object B = new Object();  // B.refCount = 1
  A.ref = B;                // B.refCount = 2
  B.ref = A;                // A.refCount = 2
  A = null;                 // A.refCount = 1 (不为0，无法回收!)
  B = null;                 // B.refCount = 1 (不为0，无法回收!)
```

#### 可达性分析法（GC Roots）

```
原理：从 GC Roots 出发，遍历对象引用链
优点：解决循环引用问题
缺点：需要 STW 停顿

GC Roots 包括：
  1. 虚拟机栈中引用的对象（局部变量）
  2. 方法区中类静态属性引用的对象
  3. 方法区中常量引用的对象
  4. 本地方法栈中 JNI 引用的对象
  5. JVM 内部引用（基本类型对应的 Class 对象、系统类加载器）
  6. 被同步锁（synchronized）持有的对象
  7. JMXBean、JVMTI 中注册的回调

示例：
      GC Root (局部变量)
           │
           ▼
      ┌─────────┐
      │ Object A │ ← 存活
      └─────────┘
           │
           ▼
      ┌─────────┐
      │ Object B │ ← 存活
      └─────────┘

      ┌─────────┐
      │ Object C │ ← 垃圾（不可达）
      └─────────┘
```

### 1.3 对象引用类型

| 引用类型 | 说明 | GC 行为 |
|---------|------|---------|
| 强引用 (Strong) | `Object obj = new Object()` | 不会被回收 |
| 软引用 (Soft) | `SoftReference<Object>` | 内存不足时回收 |
| 弱引用 (Weak) | `WeakReference<Object>` | 下次 GC 时回收 |
| 虚引用 (Phantom) | `PhantomReference<Object>` | 随时可能回收 |

## 二、Serial 收集器

### 2.1 基本特性

| 特性 | 说明 |
|------|------|
| 线程模型 | 单线程 |
| 适用区域 | 新生代 + 老年代 |
| 回收算法 | 新生代复制 + 老年代标记-整理 |
| 适用场景 | 客户端程序、小堆 (< 100MB) |
| 启用参数 | `-XX:+UseSerialGC` |
| 特点 | 简单高效、STW 时间长 |

### 2.2 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                    Serial GC 工作流程                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Young GC (Minor GC)                                       │
│   ┌─────────────────────────────────────────┐               │
│   │  1. STW (Stop-The-World)                │               │
│   │     - 暂停所有应用线程                   │               │
│   │                                          │               │
│   │  2. 单线程标记存活对象                    │               │
│   │     - 从 GC Roots 开始遍历               │               │
│   │     - 标记所有可达对象                    │               │
│   │                                          │               │
│   │  3. 复制存活对象到 Survivor               │               │
│   │     - Eden + From Survivor → To Survivor  │               │
│   │                                          │               │
│   │  4. 清空 Eden + From Survivor            │               │
│   │                                          │               │
│   │  5. 恢复应用线程                         │               │
│   └─────────────────────────────────────────┘               │
│                         ↓                                   │
│   Full GC (Major GC)                                        │
│   ┌─────────────────────────────────────────┐               │
│   │  1. STW                                  │               │
│   │                                          │               │
│   │  2. 标记所有存活对象（全堆）              │               │
│   │                                          │               │
│   │  3. 整理老年代（消除内存碎片）            │               │
│   │     - 移动存活对象                        │               │
│   │     - 更新所有引用                        │               │
│   │                                          │               │
│   │  4. 恢复应用线程                         │               │
│   └─────────────────────────────────────────┘               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.4 关键参数

```bash
# ==================== Serial GC 参数 ====================

# 启用 Serial GC
-XX:+UseSerialGC

# 新生代大小
-XX:NewSize=256m                    # 新生代初始大小
-XX:MaxNewSize=256m                 # 新生代最大大小
-Xmn256m                            # 等价于上面两个参数

# Survivor 比例
-XX:SurvivorRatio=8                 # Eden:S0:S1 = 8:1:1

# 晋升阈值
-XX:MaxTenuringThreshold=15         # 晋升老年代的年龄阈值 (0-15)

# 大对象阈值
-XX:PretenureSizeThreshold=1m       # 大于此值直接进入老年代

# GC 日志
-Xlog:gc*:file=gc.log:time,uptime,level,tags
```

### 2.5 源码分析

```cpp
// hotspot/src/share/vm/gc/serial/serialHeap.cpp

void SerialHeap::do_collection(bool full, 
                               bool clear_all_soft_refs,
                               size_t size, 
                               bool is_tlab) {
    // 1. STW
    pause();
    
    // 2. 标记阶段 (单线程)
    if (full) {
        // Full GC: 标记整个堆
        mark_live_objects_in_old_gen();
    } else {
        // Young GC: 只标记新生代
        mark_live_objects_in_young_gen();
    }
    
    // 3. 复制/整理阶段
    if (full) {
        // 老年代: 标记-整理算法
        compact_old_generation();
    } else {
        // 新生代: 复制算法
        copy_survivors();
    }
    
    // 4. 更新统计信息
    update_gc_stats();
    
    // 5. 恢复应用线程
    resume();
}

// 标记存活对象
void SerialHeap::mark_live_objects() {
    // 使用栈模拟递归
    Stack<oop> mark_stack;
    
    // 压入 GC Roots
    for (OopClosure* root_closure : root_closures()) {
        root_closure->do_oop(&mark_stack);
    }
    
    // 遍历标记
    while (!mark_stack.is_empty()) {
        oop obj = mark_stack.pop();
        if (obj != NULL && !obj->is_marked()) {
            obj->set_marked();
            // 压入子引用
            obj->oop_iterate(&mark_stack);
        }
    }
}
```

### 2.6 适用场景

### 3.3 线程数计算

```cpp
// GC 线程数计算逻辑
uint ParallelGCThreads::calc_default_active_workers(uint total_workers) {
    // 根据 CPU 核心数计算
    if (total_workers <= 8) {
        return total_workers;
    } else {
        // 超过8核时，按比例减少
        return 8 + (total_workers - 8) * 5 / 8;
    }
}

// 示例：
// 4 核  → 4 个 GC 线程
// 8 核  → 8 个 GC 线程
// 16 核 → 13 个 GC 线程
// 32 核 → 23 个 GC 线程
// 64 核 → 43 个 GC 线程
```

### 3.4 自适应调整策略

```
┌─────────────────────────────────────────────────────────────┐
│                  自适应堆大小调整                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  目标函数：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  吞吐量 = 应用运行时间 / (应用运行时间 + GC时间)      │   │
│  │                                                      │   │
│  │  目标：吞吐量 >= GCTimeRatio / (1 + GCTimeRatio)     │   │
│  │  默认：GCTimeRatio = 99，即目标吞吐量 99%             │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  调整逻辑：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  if (GC时间占比 > 目标) {                            │   │
│  │      // 吞吐量不足                                   │   │
│  │      增大堆内存                                      │   │
│  │      增大新生代比例                                  │   │
│  │      减少 GC 频率                                   │   │
│  │  }                                                   │   │
│  │                                                      │   │
│  │  if (停顿时间 > MaxGCPauseMillis) {                  │   │
│  │      // 停顿时间过长                                 │   │
│  │      减小新生代大小                                  │   │
│  │      增加 GC 频率                                   │   │
│  │  }                                                   │   │
│  │                                                      │   │
│  │  两个目标冲突时，优先满足停顿时间目标                 │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  调整的参数：                                                │
│  • NewSize / MaxNewSize（新生代大小）                        │
│  • OldSize（老年代大小）                                     │
│  • Eden 和 Survivor 的比例                                   │
│  • 晋升阈值                                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.5 Parallel Compaction 详解

```
┌─────────────────────────────────────────────────────────────┐
│              Parallel Compaction (并行整理)                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  目标：将老年代中存活对象整理到一起，消除碎片                 │
│                                                             │
│  阶段1：并行计算新位置 (Parallel Phase: Calculate Addresses) │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  ┌────┬────┬────┬────┬────┬────┬────┬────┐         │   │
│  │  │ A  │ 空 │ B  │ 空 │ C  │ 空 │ D  │ 空 │         │   │
│  │  └────┴────┴────┴────┴────┴────┴────┴────┘         │   │
│  │       ↑        ↑        ↑        ↑                  │   │
│  │      空洞     空洞     空洞     空洞                 │   │
│  │                                                      │   │
│  │  每个线程负责不同区域：                               │   │
│  │  T1: 计算 A, B 的新位置 → 地址 0, 1                  │   │
│  │  T2: 计算 C, D 的新位置 → 地址 2, 3                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  阶段2：并行移动对象 (Parallel Phase: Adjust Pointers)      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  ┌──────────────────────────────────────────────┐   │   │
│  │  │  T1: 移动 A → 地址0, B → 地址1               │   │   │
│  │  │  T2: 移动 C → 地址2, D → 地址3               │   │   │
│  │  └──────────────────────────────────────────────┘   │   │
│  │                                                      │   │
│  │  结果：                                              │   │
│  │  ┌────┬────┬────┬────┬────┬────┬────┬────┐         │   │
│  │  │ A  │ B  │ C  │ D  │ 空 │ 空 │ 空 │ 空 │         │   │
│  │  └────┴────┴────┴────┴────┴────┴────┴────┘         │   │
│  │    ↑ 连续空间，无碎片                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  阶段3：更新引用 (Parallel Phase: Update References)        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 并行更新所有指向旧地址的引用为新地址               │   │
│  │  • 包括栈、堆、方法区中的引用                         │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 四、CMS 收集器

### 4.1 基本特性

| 特性 | 说明 |
|------|------|
| 线程模型 | 并发标记、并行清除 |
| 适用区域 | 老年代（配合 ParNew 收集新生代） |
| 回收算法 | 标记-清除（不整理，有碎片） |
| 适用场景 | Web 服务器、响应时间敏感应用 |
| 启用参数 | `-XX:+UseConcMarkSweepGC` |
| 特点 | 低停顿、有碎片、已废弃 (JDK 14 移除) |

### 4.2 工作流程（7 个阶段）

```
┌─────────────────────────────────────────────────────────────┐
│                     CMS 工作流程                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  阶段1: 初始标记 (Initial Mark) - STW ⚠️                    │
│  ┌─────────────────────────────────────────┐               │
│  │  • 标记 GC Roots 直接引用的对象            │               │
│  │  • 速度很快 (借助 OopMap)                 │               │
│  │  • 停顿时间: ~1-10ms                      │               │
│  │                                          │               │
│  │  标记内容:                                │               │
│  │  - 虚拟机栈中引用的对象                   │               │
│  │  - 静态属性引用的对象                     │               │
│  │  - 常量引用的对象                         │               │
│  │  - JNI 引用的对象                         │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段2: 并发标记 (Concurrent Mark) - 并发 ✅                 │
│  ┌─────────────────────────────────────────┐               │
│  │  • 从 GC Roots 遴历整个对象图              │               │
│  │  • 与应用线程并发执行                      │               │
│  │  • 耗时最长的阶段（可能数秒）              │               │
│  │  • 使用三色标记算法                        │               │
│  │                                          │               │
│  │  问题: 并发期间引用可能变化                │               │
│  │  解决: 写屏障 + Incremental Update        │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段3: 并发预清理 (Concurrent Preclean) - 并发 ✅           │
│  ┌─────────────────────────────────────────┐               │
│  │  • 处理并发标记期间新增的引用              │               │
│  │  • 减少下一阶段 STW 时间                  │               │
│  │  • 扫描 Dirty Card (Card Table)          │               │
│  │                                          │               │
│  │  Card Table:                             │               │
│  │  ┌────┬────┬────┬────┬────┐             │               │
│  │  │ 0  │ 1  │ 0  │ 1  │ 0  │             │               │
│  │  └────┴────┴────┴────┴────┘             │               │
│  │    ↑        ↑                            │               │
│  │   干净     脏卡（有跨代引用）             │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段4: 可中断预清理 (Concurrent Abortable Preclean) - 并发 │
│  ┌─────────────────────────────────────────┐               │
│  │  • 等待 Young GC 发生                     │               │
│  │  • 避免 Young GC 和 Remark 同时发生       │               │
│  │  • 可设置退出条件                         │               │
│  │                                          │               │
│  │  为什么等待 Young GC?                     │               │
│  │  - Young GC 会扫描新生代引用              │               │
│  │  - 减少 Remark 阶段需要扫描的对象         │               │
│  │                                          │               │
│  │  退出条件:                                │               │
│  │  - 循环次数达到阈值                       │               │
│  │  - 老年代使用率达到阈值                   │               │
│  │  - Young GC 发生                          │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段5: 重新标记 (Remark) - STW ⚠️                          │
│  ┌─────────────────────────────────────────┐               │
│  │  • 修正并发标记期间变化的引用              │               │
│  │  • 比初始标记慢，但比并发标记快            │               │
│  │  • 停顿时间: ~10-50ms                     │               │
│  │                                          │               │
│  │  处理内容:                                │               │
│  │  - 重新扫描 GC Roots                      │               │
│  │  - 处理并发标记期间新增的引用              │               │
│  │  - 处理并发标记期间删除的引用              │               │
│  │                                          │               │
│  │  优化手段:                                │               │
│  │  - -XX:+CMSParallelRemarkEnabled          │               │
│  │  - -XX:+CMSScavengeBeforeRemark           │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段6: 并发清除 (Concurrent Sweep) - 并发 ✅               │
│  ┌─────────────────────────────────────────┐               │
│  │  • 清除未标记的垃圾对象                    │               │
│  │  • 与应用线程并发执行                      │               │
│  │  • 使用空闲列表 (Free List)               │               │
│  │                                          │               │
│  │  清除方式:                                │               │
│  │  - 遍历老年代所有对象                     │               │
│  │  - 未标记的对象加入 Free List              │               │
│  │  - 不移动对象（不整理）                   │               │
│  │                                          │               │
│  │  问题: 产生内存碎片                       │               │
│  │  解决: 定期 Full GC 整理                  │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段7: 并发重置 (Concurrent Reset) - 并发 ✅               │
│  ┌─────────────────────────────────────────┐               │
│  │  • 重置 CMS 内部数据结构                   │               │
│  │  • 清空标记位图                           │               │
│  │  • 为下次 GC 做准备                       │               │
│  └─────────────────────────────────────────┘               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 三色标记算法

```
┌─────────────────────────────────────────────────────────────┐
│                    三色标记算法                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  三种颜色的含义：                                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  白色 (White):                                      │   │
│  │  - 尚未被扫描的对象                                 │   │
│  │  - GC 结束后仍为白色 → 为垃圾，将被回收              │   │
│  │                                                      │   │
│  │  灰色 (Gray):                                       │   │
│  │  - 已被扫描，但子对象未扫描                          │   │
│  │  - 对象本身已标记为存活                              │   │
│  │  - 其引用的子对象尚未处理                            │   │
│  │                                                      │   │
│  │  黑色 (Black):                                      │   │
│  │  - 已被扫描，且子对象也已扫描                        │   │
│  │  - 对象及其所有引用都已处理完毕                      │   │
│  │  - 不会再被重复扫描                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  标记过程：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │        [GC Root]                                     │   │
│  │            │                                         │   │
│  │            ▼                                         │   │
│  │     ┌──────────┐                                    │   │
│  │     │  灰色 A   │ ← 正在处理                        │   │
│  │     └──────────┘                                    │   │
│  │            │                                         │   │
│  │            ▼                                         │   │
│  │     ┌──────────┐    ┌──────────┐                   │   │
│  │     │  黑色 B   │───→│  白色 C   │                   │   │
│  │     └──────────┘    └──────────┘                   │   │
│  │          ↑               ↑                          │   │
│  │        已完成          待处理                        │   │
│  │                                                      │   │
│  │  过程：                                              │   │
│  │  1. GC Root 标记为灰色，压入灰色栈                   │   │
│  │  2. 从灰色栈取出对象 A，扫描其引用                   │   │
│  │  3. A 引用 B，将 B 标记为灰色，压入栈                │   │
│  │  4. A 处理完毕，标记为黑色                           │   │
│  │  5. 取出 B，扫描其引用                               │   │
│  │  6. B 引用 C，将 C 标记为灰色，压入栈                │   │
│  │  7. B 处理完毕，标记为黑色                           │   │
│  │  8. 取出 C，C 无引用，标记为黑色                     │   │
│  │  9. 栈为空，标记完成                                 │   │
│  │                                                      │   │
│  │  最终：白色对象为垃圾                                 │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.4 并发标记漏标问题

```
┌─────────────────────────────────────────────────────────────┐
│              漏标条件 (两种情况同时发生)                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  条件1: 黑色对象新增了对白色对象的引用                        │
│  ─────────────────────────────────────                      │
│         [黑A] ──────→ [白C]   (新增引用)                    │
│                                                             │
│  条件2: 所有灰色对象到白色对象的可达路径被切断                 │
│  ─────────────────────────────────────                      │
│         [灰B] ──✕──→ [白C]   (删除引用)                    │
│                                                             │
│  结果: 白色 C 被误判为垃圾 (漏标)                            │
│                                                             │
│  原因分析：                                                  │
│  • 黑色 A 不会再被扫描（已经处理完毕）                       │
│  • 灰色 B 到 C 的引用已删除                                 │
│  • C 既不会被 A 扫描，也不会被 B 扫描                       │
│  • C 最终被误判为白色（垃圾）而回收                          │
│                                                             │
│  这会导致严重问题：                                          │
│  • 存活对象被错误回收                                       │
│  • 程序出现悬空指针                                         │
│  • 可能导致程序崩溃                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    解决方案: 写屏障                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  CMS 使用 Incremental Update (增量更新)：                    │
│  ─────────────────────────────────────                      │
│                                                             │
│  原理：                                                      │
│  • 当黑色对象新增对白色对象的引用时                          │
│  • 写屏障记录这个变化                                       │
│  • 将黑色对象重新标记为灰色（重新扫描）                      │
│                                                             │
│  伪代码：                                                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  void write_barrier(Object obj, Object ref) {       │   │
│  │      obj.field = ref;                               │   │
│  │      if (obj.color == BLACK && ref.color == WHITE) {│   │
│  │          obj.color = GRAY;  // 重新标记为灰色       │   │
│  │          remark_queue.add(obj);                     │   │
│  │      }                                              │   │
│  │  }                                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  处理时机：                                                  │
│  • Remark 阶段处理 remark_queue 中的对象                    │
│  • 重新扫描这些对象，确保不会漏标                            │
│                                                             │
│  缺点：                                                      │
│  • Remark 阶段需要重新扫描灰色对象的子对象                   │
│  • 可能导致 Remark 停顿时间较长                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.5 Card Table (卡表)

```
┌─────────────────────────────────────────────────────────────┐
│                      Card Table                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  作用：跟踪老年代对新生代的引用（跨代引用）                    │
│                                                             │
│  问题背景：                                                  │
│  • Young GC 只回收新生代                                     │
│  • 但老年代可能引用了新生代对象                              │
│  • 如何快速找到这些跨代引用？                                │
│  • 不可能每次 Young GC 都扫描整个老年代                      │
│                                                             │
│  解决方案：Card Table                                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  老年代内存（每 512 字节对应一个 Card）               │   │
│  │  ┌─────────────────────────────────────────────┐    │   │
│  │  │ [512B][512B][512B][512B][512B][512B][512B]   │    │   │
│  │  │   0     1     2     3     4     5     6     │    │   │
│  │  └─────────────────────────────────────────────┘    │   │
│  │                         ↓                            │   │
│  │  Card Table（1 byte per 512 bytes heap）             │   │
│  │  ┌─────────────────────────────────────────────┐    │   │
│  │  │ [0] [1] [0] [0] [1] [0] [0]                 │    │   │
│  │  │  ↑       ↑                                   │    │   │
│  │  │ 干净    脏卡（有跨代引用）                    │    │   │
│  │  └─────────────────────────────────────────────┘    │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  写屏障实现：                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  void write_barrier(Object obj, Object ref) {       │   │
│  │      obj.field = ref;                               │   │
│  │      // 如果是跨代引用，标记对应的 Card 为脏         │   │
│  │      if (is_old_gen(obj) && is_young_gen(ref)) {    │   │
│  │          card_table[obj >> 9] = DIRTY;  // 512=2^9  │   │
│  │      }                                              │   │
│  │  }                                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  Young GC 时：                                               │
│  • 只扫描脏 Card 对应的老年代区域                           │
│  • 不需要扫描整个老年代                                     │
│  • 大幅减少扫描范围                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.6 并发模式失败

```
┌─────────────────────────────────────────────────────────────┐
│              并发模式失败 (Concurrent Mode Failure)          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  触发条件：                                                  │
│  • 老年代空间不足以容纳新生代晋升的对象                       │
│  • CMS 并发清除未完成，新对象持续分配                        │
│                                                             │
│  发生过程：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  老年代: ████████████░░░░  (80%)                     │   │
│  │                     ↑                                │   │
│  │              CMS 正在并发清除                         │   │
│  │                     │                                │   │
│  │  新生代: ████████████  (需要晋升)                    │   │
│  │                     ↓                                │   │
│  │              空间不足! CMF!                          │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  后果：                                                      │
│  • 退化为 Serial Old GC                                    │
│  • 单线程标记-整理                                         │
│  • 长时间 STW（可能数秒甚至更长）                           │
│  • 性能急剧下降                                            │
│                                                             │
│  日志示例：                                                  │
│  [GC (Allocation Failure) [ParNew: 349440K->43520K(393216K), 0.0234567 secs] |
│  [CMS: 2097152K->2097151K(2097152K), 1.2345678 secs]       │
│  [CMS-initial-mark: 2097152K(2097152K)]                     │
│                                                             │
│  预防措施：                                                  │
│  1. 降低触发阈值                                            │
│     -XX:CMSInitiatingOccupancyFraction=60                   │
│                                                              │
│  2. 增大老年代空间                                           │
│     -Xmx4g -XX:NewRatio=3                                  │
│                                                              │
│  3. 减少对象分配速率                                         │
│     - 优化代码，减少临时对象                                 │
│                                                              │
│  4. 增加并发线程                                             │
│     -XX:ConcGCThreads=8                                     │
│                                                              │
│  5. 提前触发 CMS                                             │
│     -XX:+UseCMSInitiatingOccupancyOnly                      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 4.10 CMS 废弃原因

```

├─────────────────────────────────────────────────────────────┤

```

## 五、G1 收集器

### 5.1 基本特性

| 特性 | 说明 |
|------|------|
| 线程模型 | 并发标记、并行回收 |
| 适用区域 | 整个堆（Region 化） |
| 回收算法 | 整体标记-整理，局部复制 |
| 适用场景 | 大堆 (6GB+)、低延迟、混合负载 |
| 启用参数 | `-XX:+UseG1GC` (JDK 9+ 默认) |
| 特点 | Region 化、可预测停顿、无碎片 |

### 5.2 Region 内存布局

```
┌─────────────────────────────────────────────────────────────┐
│                     G1 内存布局                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  堆内存被划分为多个大小相等的 Region（1MB-32MB）             │
│                                                              │
│  ┌────┬────┬────┬────┬────┬────┬────┬────┐                 │
│  │ E  │ E  │ S  │ O  │ O  │ H  │ H  │    │                 │
│  ├────┼────┼────┼────┼────┼────┼────┼────┤                 │
│  │ O  │ E  │    │ O  │ E  │ S  │ O  │ E  │                 │
│  ├────┼────┼────┼────┼────┼────┼────┼────┤                 │
│  │ E  │ O  │ E  │    │ O  │ E  │ O  │ E  │                 │
│  └────┴────┴────┴────┴────┴────┴────┴────┘                 │
│                                                              │
│  Region 类型：                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  E = Eden Region     新生代 Eden 区                  │   │
│  │  S = Survivor Region 新生代 Survivor 区              │   │
│  │  O = Old Region      老年代                          │   │
│  │  H = Humongous Region 大对象（> Region 50%）         │   │
│  │  空白 = 空闲 Region                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  Region 大小计算：                                           │
│  • 堆大小 / Region 数量                                     │
│  • 每个 Region 1MB-32MB（2的幂次）                          │
│  • 堆越大，Region 越大                                      │
│                                                              │
│  示例：                                                      │
│  • 4GB 堆 → 2048 个 Region，每个 2MB                        │
│  • 16GB 堆 → 2048 个 Region，每个 8MB                       │
│                                                              │
│  Humongous 对象：                                            │
│  • 大小 > Region 的 50%                                     │
│  • 分配在连续的 Humongous Region 中                         │
│  • 直接进入老年代                                           │
│  • 不参与 Young GC                                          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 5.3 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                     G1 工作流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Young GC（纯新生代回收）                                     │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  触发条件：Eden 区满                      │               │
│  │                                          │               │
│  │  步骤：                                   │               │
│  │  1. STW                                  │               │
│  │  2. 并行扫描 Eden 和 Survivor             │               │
│  │  3. 复制存活对象到新的 Survivor            │               │
│  │  4. 年龄达到阈值的对象晋升到 Old          │               │
│  │  5. 清空 Eden 和原 Survivor               │               │
│  │  6. 恢复应用线程                         │               │
│  │                                          │               │
│  │  特点：                                   │               │
│  │  • 只回收新生代                           │               │
│  │  • 停顿时间短                             │               │
│  │  • 高频率触发                             │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                         ↓                                   │
│  并发标记周期（Concurrent Marking Cycle）                    │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  触发条件：                               │               │
│  │  • 堆使用率达到阈值（默认 45%）           │               │
│  │  • -XX:InitiatingHeapOccupancyPercent=45  │               │
│  │                                          │               │
│  │  阶段1: 初始标记（Initial Mark）- STW    │               │
│  │  ┌─────────────────────────────────────┐ │               │
│  │  │ • 标记 GC Roots                     │ │               │
│  │  │ • 借助 Young GC 完成（复用停顿）     │ │               │
│  │  │ • 不产生额外停顿                     │ │               │
│  │  └─────────────────────────────────────┘ │               │
│  │                     ↓                     │               │
│  │  阶段2: 并发标记（Concurrent Marking）   │               │
│  │  ┌─────────────────────────────────────┐ │               │
│  │  │ • 遍历对象图                         │ │               │
│  │  │ • 标记所有存活对象                    │ │               │
│  │  │ • 与应用线程并发执行                  │ │               │
│  │  │ • 使用 SATB 写屏障                   │ │               │
│  │  └─────────────────────────────────────┘ │               │
│  │                     ↓                     │               │
│  │  阶段3: 最终标记（Final Mark）- STW     │               │
│  │  ┌─────────────────────────────────────┐ │               │
│  │  │ • 处理 SATB 缓冲区                  │ │               │
│  │  │ • 完成所有存活对象标记                │ │               │
│  │  │ • 停顿时间：~10-50ms                │ │               │
│  │  └─────────────────────────────────────┘ │               │
│  │                     ↓                     │               │
│  │  阶段4: 筛选回收（Cleanup）             │               │
│  │  ┌─────────────────────────────────────┐ │               │
│  │  │ • 统计每个 Region 存活对象比例       │ │               │
│  │  │ • 计算回收价值（垃圾比例）           │ │               │
│  │  │ • 排序回收价值                       │ │               │
│  │  │ • 选择回收集合（Collection Set）     │ │               │
│  │  │ • 完全空闲的 Region 立即回收         │ │               │
│  │  └─────────────────────────────────────┘ │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                         ↓                                   │
│  Mixed GC（混合回收）                                        │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  触发条件：并发标记完成后                 │               │
│  │                                          │               │
│  │  步骤：                                   │               │
│  │  1. STW                                  │               │
│  │  2. 回收所有 Eden 和 Survivor             │               │
│  │  3. 回收部分垃圾比例高的 Old Region       │               │
│  │  4. 复制存活对象，整理内存碎片            │               │
│  │  5. 恢复应用线程                         │               │
│  │                                          │               │
│  │  选择策略：                               │               │
│  │  • 优先回收垃圾比例高的 Region            │               │
│  │  • 确保停顿时间不超过目标                 │               │
│  │  • 多次 Mixed GC 完成老年代回收           │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                         ↓                                   │
│  Full GC（兜底回收）                                         │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  触发条件（尽量避免）：                   │               │
│  │  • Mixed GC 来不及回收                    │               │
│  │  • 晋升失败                               │               │
│  │  • 大对象分配失败                         │               │
│  │                                          │               │
│  │  特点：                                   │               │
│  │  • 单线程（JDK 10 之前）                 │               │
│  │  • 长时间 STW                             │               │
│  │  • 应该尽量避免                           │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.4 SATB (Snapshot-At-The-Beginning)

```
┌─────────────────────────────────────────────────────────────┐
│              SATB 写屏障机制                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  核心思想：                                                  │
│  • 并发标记开始时，记录对象图的"快照"                        │
│  • 并发期间删除的引用，不要立即清理                          │
│  • 确保标记开始时存活的对象，在标记结束时仍被标记             │
│  • 宁可多标（浮动垃圾），不可漏标                            │
│                                                              │
│  实现原理：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  void write_barrier(Object obj, Object old_ref) {   │   │
│  │      obj.field = new_ref;                           │   │
│  │      // 删除引用时，将旧引用记录到 SATB 缓冲区       │   │
│  │      if (old_ref != null) {                         │   │
│  │          satb_buffer.enqueue(old_ref);              │   │
│  │      }                                              │   │
│  │  }                                                  │   │
│  │                                                      │   │
│  │  注意：只在引用被删除时触发，新增引用不处理           │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  缓冲区处理：                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  1. 线程本地缓冲区（Thread Local Buffer）             │   │
│  │     • 每个线程独立的 SATB 缓冲区                     │   │
│  │     • 减少锁竞争                                    │   │
│  │                                                      │   │
│  │  2. 全局缓冲区列表                                   │   │
│  │     • 线程缓冲区满后放入全局列表                     │   │
│  │     • 并发标记线程定期处理                           │   │
│  │                                                      │   │
│  │  3. 最终标记阶段处理                                 │   │
│  │     • STW 时处理所有剩余缓冲区                       │   │
│  │     • 确保不漏标                                    │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  对比 CMS 的 Incremental Update：                           │
│  ┌─────────────────┬─────────────────────────────────────┐ │
│  │                 │ CMS              │ G1               │ │
│  ├─────────────────┼──────────────────┼──────────────────┤ │
│  │ 写屏障触发条件  │ 新增引用时       │ 删除引用时       │ │
│  │ 处理方式        │ 重新标记为灰色   │ 记录到 SATB 缓冲 │ │
│  │ Remark 停顿    │ 较长（需重新扫描）│ 较短（处理缓冲） │ │
│  │ 浮动垃圾        │ 较少             │ 较多             │ │
│  │ 实现复杂度      │ 较高             │ 较低             │ │
│  └─────────────────┴──────────────────┴──────────────────┘ │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.5 RSet (Remembered Set)

```
┌─────────────────────────────────────────────────────────────┐
│                RSet (记忆集)                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  作用：记录 Region 之间的引用关系                            │
│                                                              │
│  问题背景：                                                  │
│  • G1 以 Region 为单位回收                                   │
│  • 回收某个 Region 时，需要知道有哪些 Region 引用了它        │
│  • 不可能扫描整个堆来查找                                    │
│                                                              │
│  解决方案：RSet                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  每个 Region 维护一个 RSet                             │   │
│  │                                                      │   │
│  │  Region A 的 RSet:                                   │   │
│  │  ┌───────────────────────────────────────────────┐ │   │
│  │  │  Card Table（粗粒度）                          │ │   │
│  │  │  ┌────┬────┬────┬────┬────┐                  │ │   │
│  │  │  │ 0  │ 1  │ 0  │ 1  │ 0  │                  │ │   │
│  │  │  └────┴────┴────┴────┴────┘                  │ │   │
│  │  │      ↑        ↑                              │ │   │
│  │  │   Region B  Region C 引用了 A                 │ │   │
│  │  └───────────────────────────────────────────────┘ │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  数据结构（PRT - Per Region Table）：                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  1. 稀疏表（Sparse Table）                           │   │
│  │     • 使用 HashMap 存储                              │   │
│  │     • 适用于引用较少的情况                           │   │
│  │                                                      │   │
│  │  2. 细粒度位图（Fine Bitmap）                        │   │
│  │     • 每个 Card 对应一个 bit                         │   │
│  │     • 适用于引用较多的情况                           │   │
│  │                                                      │   │
│  │  3. 粗粒度位图（Coarse Bitmap）                      │   │
│  │     • 每个 Region 对应一个 bit                       │   │
│  │     • 适用于引用非常密集的情况                       │   │
│  │                                                      │   │
│  │  自动升级：Sparse → Fine → Coarse                    │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  写屏障实现：                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  void write_barrier(Object obj, Object ref) {       │   │
│  │      obj.field = ref;                               │   │
│  │      // 记录引用关系到 RSet                          │   │
│  │      Region from = get_region(obj);                 │   │
│  │      Region to = get_region(ref);                   │   │
│  │      if (from != to) {                              │   │
│  │          to.rset.add(from, obj);                    │   │
│  │      }                                              │   │
│  │  }                                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  RSet 维护成本：                                             │
│  • 写屏障开销：5%-10%                                       │
│  • 内存开销：5%-20%                                         │
│  • G1 通过控制 RSet 精度来平衡                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.6 停顿预测模型

```
┌─────────────────────────────────────────────────────────────┐
│                  停顿预测模型                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  目标：-XX:MaxGCPauseMillis=200ms（默认）                    │
│                                                              │
│  预测依据：                                                  │
│  1. 每个 Region 的垃圾比例                                  │
│     • 存活对象大小 / Region 大小                            │
│     • 垃圾比例越高，回收价值越大                            │
│                                                              │
│  2. 历史回收耗时统计                                        │
│     • 记录每次回收的实际耗时                                │
│     • 用于预测下次回收耗时                                  │
│                                                              │
│  3. 存活对象大小                                            │
│     • 需要复制的对象越多，耗时越长                          │
│                                                              │
│  选择策略：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  Region 列表（按回收价值排序）：                      │   │
│  │  ┌──────────┬──────────┬──────────┬──────────┐     │   │
│  │  │ Region 1 │ Region 2 │ Region 3 │ Region 4 │     │   │
│  │  │ 垃圾 90% │ 垃圾 85% │ 垃圾 70% │ 垃圾 60% │     │   │
│  │  │ 预计 10ms│ 预计 15ms│ 预计 20ms│ 预计 30ms│     │   │
│  │  └──────────┴──────────┴──────────┴──────────┘     │   │
│  │       ↑           ↑          ↑                       │   │
│  │    优先回收   次优先     最后                         │   │
│  │                                                      │   │
│  │  累计预计耗时：                                       │   │
│  │  Region 1: 10ms < 200ms ✓                           │   │
│  │  Region 1+2: 25ms < 200ms ✓                         │   │
│  │  Region 1+2+3: 45ms < 200ms ✓                       │   │
│  │  Region 1+2+3+4: 75ms < 200ms ✓                     │   │
│  │                                                      │   │
│  │  当累计预计耗时接近 MaxGCPauseMillis 时停止选择       │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  动态调整：                                                  │
│  • 如果实际停顿 < 目标：可以回收更多 Region                 │
│  • 如果实际停顿 > 目标：减少回收 Region                     │
│  • 自动平衡回收效率和停顿时间                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.7 关键参数

```bash
# ==================== G1 GC 参数 ====================

# 启用 G1
-XX:+UseG1GC                        # 启用 G1 (JDK 9+ 默认)

# Region 大小
-XX:G1HeapRegionSize=16m            # Region 大小 (1MB-32MB，2的幂次)

# 停顿目标
-XX:MaxGCPauseMillis=200            # 最大停顿时间目标 (ms)

# 并发线程
-XX:ConcGCThreads=4                 # 并发标记线程数
-XX:ParallelGCThreads=8             # 并行线程数

# 触发阈值
-XX:InitiatingHeapOccupancyPercent=45 # 整堆使用率触发并发标记

# Region 相关
-XX:G1ReservePercent=10             # 保留内存比例（防止晋升失败）
-XX:G1HeapWastePercent=5            # 可接受的浪费比例
-XX:G1MixedGCCountTarget=8          # Mixed GC 次数目标
-XX:G1OldCSetRegionThresholdPercent=10 # 每次 Mixed GC 回收老年代比例

# 其他
-XX:G1NewSizePercent=5              # 新生代最小比例
-XX:G1MaxNewSizePercent=60          # 新生代最大比例
-XX:G1RSetUpdatingPauseTimePercent=5 # RSet 更新暂停时间比例

# GC 日志
-Xlog:gc*:file=gc.log:time,uptime,level,tags
```

### 6.1 基本特性

| 特性 | 说明 |

### 6.2 着色指针 (Colored Pointers)

```
┌─────────────────────────────────────────────────────────────┐
│                  ZGC 着色指针                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  传统 GC：修改对象头中的标记位                               │
│  ZGC：直接在指针中嵌入标记信息                               │
│                                                              │
│  64 位指针结构：                                             │
│  ┌─────────┬─────────────────────────────────────────────┐ │
│  │ 着色位  │                对象地址                       │ │
│  │ (4 bits)│               (44 bits)                     │ │
│  ├─────────┴─────────────────────────────────────────────┤ │
│  │ 63    60│59                                       16│15  0│ │
│  │ ┌───┬───┬─────────────────────────────────────────┬─────┐│ │
│  │ │F  │R  │M0 │M1 │        对象地址                  │     ││ │
│  │ └───┴───┴───┴───┴─────────────────────────────────┴─────┘│ │
│  │  ↑   ↑   ↑   ↑                                        │ │
│  │  │   │   │   └── Marked1: 并发标记阶段 1              │ │
│  │  │   │   └────── Marked0: 并发标记阶段 0              │ │
│  │  │   └────────── Remapped: 已重映射                   │ │
│  │  └────────────── Finalizable: 可终结                  │ │
│  └─────────────────────────────────────────────────────────┘ │
│                                                              │
│  着色位的作用：                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  Marked0 / Marked1:                                  │   │
│  │  • 交替使用，标识当前标记周期                         │   │
│  │  • 标记阶段 0 使用 Marked0，阶段 1 使用 Marked1      │   │
│  │  • 避免混淆不同周期的标记                             │   │
│  │                                                      │   │
│  │  Remapped:                                           │   │
│  │  • 标识对象是否已重映射到新地址                       │   │
│  │  • 转移阶段后，需要将引用更新为新地址                 │   │
│  │                                                      │   │
│  │  Finalizable:                                        │   │
│  │  • 标识对象需要执行 finalize()                       │   │
│  │  • 需要特殊处理                                      │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  优点：                                                      │
│  • 不修改对象头，不影响应用线程                              │
│  • 并发标记/转移通过修改指针完成                             │
│  • 支持多重映射（Multiple Mapping）                          │
│                                                              │
│  缺点：                                                      │
│  • 可寻址空间减少（44位 = 16TB）                             │
│  • 需要操作系统支持多重映射                                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 多重映射

```
┌─────────────────────────────────────────────────────────────┐
│                    多重映射                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  定义：同一物理内存可以映射到多个虚拟地址                    │
│                                                              │
│  虚拟地址空间：                                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  Marked0 视图    Marked1 视图    Remapped 视图      │   │
│  │  0x1000          0x2000          0x3000             │   │
│  │     ↓               ↓               ↓               │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                     ↓           ↓           ↓               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │              同一物理页（Page）                       │   │
│  │              0x7f0000000                             │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  作用：                                                      │
│  • 对象转移时，旧地址和新地址同时有效                        │
│  • 应用线程可以继续使用旧地址                                │
│  • 读屏障自动修正为新地址                                    │
│                                                              │
│  实现原理：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  1. 转移前：                                         │   │
│  │     虚拟地址 0x1000 (Marked0) → 物理地址 A          │   │
│  │                                                      │   │
│  │  2. 转移中：                                         │   │
│  │     物理地址 A 的对象复制到物理地址 B                │   │
│  │     转发表记录：旧地址 0x1000 → 新地址 0x2000       │   │
│  │                                                      │   │
│  │  3. 转移后：                                         │   │
│  │     虚拟地址 0x1000 (Marked0) → 物理地址 A（旧）    │   │
│  │     虚拟地址 0x2000 (Marked1) → 物理地址 B（新）    │   │
│  │                                                      │   │
│  │  4. 读屏障修正：                                     │   │
│  │     访问 0x1000 → 检查着色位 → 返回 0x2000          │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  操作系统支持：                                              │
│  • Linux: mmap + MAP_SHARED                                  │
│  • Windows: VirtualAlloc                                     │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 6.4 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                     ZGC 工作流程                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  阶段1: 初始标记（Initial Mark）- STW ⚠️ (~1ms)             │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  目的：标记 GC Roots 直接引用的对象       │               │
│  │                                          │               │
│  │  操作：                                   │               │
│  │  • 扫描 GC Roots                         │               │
│  │  • 设置指针的 Marked0 位                  │               │
│  │  • 压入标记栈                             │               │
│  │                                          │               │
│  │  为什么这么快？                           │               │
│  │  • 只标记直接引用，不遍历对象图           │               │
│  │  • 借助 OopMap 快速定位                  │               │
│  │  • 极短停顿                              │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段2: 并发标记（Concurrent Mark）- 并发 ✅                │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  目的：遍历对象图，标记所有存活对象       │               │
│  │                                          │               │
│  │  操作：                                   │               │
│  │  • 从标记栈取出对象                       │               │
│  │  • 扫描对象的引用字段                     │               │
│  │  • 设置引用对象的 Marked0 位              │               │
│  │  • 压入标记栈                             │               │
│  │  • 重复直到栈为空                         │               │
│  │                                          │               │
│  │  并发安全：                               │               │
│  │  • 使用读屏障处理并发访问                 │               │
│  │  • 应用线程读取引用时，检查着色位         │               │
│  │  • 如果未标记，帮助标记                   │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段3: 并发预备重分配（Concurrent Prepare for Relocation） │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  目的：选择需要转移的 Page                │               │
│  │                                          │               │
│  │  操作：                                   │               │
│  │  • 统计每个 Page 的存活对象               │               │
│  │  • 计算每个 Page 的垃圾比例               │               │
│  │  • 选择回收价值高的 Page                  │               │
│  │  • 组成 Relocation Set                    │               │
│  │                                          │               │
│  │  选择策略：                               │   │
│  │  • 优先回收垃圾比例高的 Page              │   │
│  │  • 确保有足够的空闲 Page 接收对象         │   │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段4: 并发重分配（Concurrent Relocation）- 并发 ✅        │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  目的：将存活对象转移到新 Page            │               │
│  │                                          │               │
│  │  操作：                                   │               │
│  │  • 分配新的空闲 Page                      │               │
│  │  • 将 Relocation Set 中的对象复制到新 Page│               │
│  │  • 在转发表（Forwarding Table）记录映射   │               │
│  │                                          │               │
│  │  转发表结构：                             │               │
│  │  ┌─────────────────────────────────────┐ │               │
│  │  │ 旧地址  →  新地址                   │ │               │
│  │  │ 0x1000  →  0x5000                   │ │               │
│  │  │ 0x1008  →  0x5008                   │ │               │
│  │  │ ...                                 │ │               │
│  │  └─────────────────────────────────────┘ │               │
│  │                                          │               │
│  │  并发安全：                               │               │
│  │  • 读屏障检查对象是否已转移               │               │
│  │  • 如果已转移，自动重定向到新地址         │               │
│  │  • 应用线程无感知                         │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段5: 并发重映射（Concurrent Remap）- 并发 ✅             │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  目的：修正所有引用为新地址               │               │
│  │                                          │               │
│  │  操作：                                   │               │
│  │  • 遍历所有引用                           │               │
│  │  • 检查是否指向已转移的对象               │               │
│  │  • 更新为新地址                           │               │
│  │  • 更新 Remapped 位                       │               │
│  │                                          │               │
│  │  为什么需要这个阶段？                     │               │
│  │  • 读屏障有性能开销                       │               │
│  │  • 提前修正可以减少后续读屏障开销         │               │
│  │  • 为下次 GC 做准备                       │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                                                             │
│  总停顿时间：~1-2ms（仅初始标记阶段）                        │
│  其他阶段完全并发，不影响应用                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.5 读屏障 (Load Barrier)

```cpp
// ZGC 读屏障伪代码
Object* load_barrier(Object* obj) {
    // 检查指针着色位
    if (obj->color == MARKED0 || obj->color == MARKED1) {
        // 对象已被标记或正在转移
        if (obj->is_forwarded()) {
            // 已转移，返回新地址
            return obj->forwarding_address();
        } else {
            // 未转移，尝试转移
            return try_relocate(obj);
        }
    }
    return obj;  // 正常对象，直接返回
}

// JIT 编译后插入读屏障
void access_field(Object* obj, int offset) {
    Object* ref = obj->fields[offset];
    ref = load_barrier(ref);  // 读屏障
    obj->fields[offset] = ref;
}

// 读屏障的开销
// • 每次访问引用字段都需要检查
// • JIT 可以优化：只在必要时插入屏障
// • 总体开销：~1%-5%
```

## 七、Shenandoah 收集器

### 7.1 基本特性

| 特性 | 说明 |
|------|------|
| 线程模型 | 并发执行 |
| 适用区域 | 整个堆 |
| 回收算法 | 并发标记-整理 |
| 适用场景 | 低延迟要求、大堆 |
| 启用参数 | `-XX:+UseShenandoahGC` |
| 特点 | 亚毫秒停顿、转发指针 |

### 7.2 转发指针 (Forwarding Pointer)

```
┌─────────────────────────────────────────────────────────────┐
│                   转发指针                                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  原理：每个对象头部额外增加一个指针（8 字节）                │
│                                                              │
│  对象结构：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                   Object Header                     │   │
│  │  ┌──────────┬──────────┬──────────┐                │   │
│  │  │Mark Word │ Klass Ptr│Fwd Ptr   │                │   │
│  │  │ (8 bytes)│ (8 bytes)│ (8 bytes)│                │   │
│  │  └──────────┴──────────┴──────────┘                │   │
│  │                       ↑                             │   │
│  │                  转发指针                            │   │
│  │                  • 指向自己（默认）                  │   │
│  │                  • 指向新位置（转移后）              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  转移过程：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  1. 初始状态：                                       │   │
│  │     Object A @ 0x1000                                │   │
│  │     fwd_ptr = 0x1000 (指向自己)                      │   │
│  │                                                      │   │
│  │  2. 分配新空间：                                     │   │
│  │     Object A' @ 0x5000 (新位置)                      │   │
│  │                                                      │   │
│  │  3. 复制对象：                                       │   │
│  │     复制 Object A 的内容到 0x5000                    │   │
│  │                                                      │   │
│  │  4. 更新转发指针（CAS）：                            │   │
│  │     Object A @ 0x1000                                │   │
│  │     fwd_ptr = 0x5000 (指向新位置)                    │   │
│  │                                                      │   │
│  │  5. 应用线程访问：                                   │   │
│  │     访问 0x1000 → 读取 fwd_ptr → 跳转到 0x5000      │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  优点：                                                      │
│  • 无需读屏障（相比 ZGC）                                   │
│  • 实现相对简单                                             │
│  • 转发逻辑直观                                             │
│                                                              │
│  缺点：                                                      │
│  • 每个对象增加 8 字节开销                                  │
│  • 需要额外的 CAS 操作保证并发安全                          │
│  • 对象头变大，可能影响缓存效率                              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 7.3 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                Shenandoah 工作流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  阶段1: 初始标记（Init Mark）- STW ⚠️                       │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  目的：标记 GC Roots                      │               │
│  │                                          │               │
│  │  操作：                                   │               │
│  │  • 扫描 GC Roots                         │               │
│  │  • 标记直接引用的对象                     │               │
│  │  • 停顿时间：~1ms                        │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段2: 并发标记（Concurrent Mark）- 并发 ✅                │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  目的：遍历对象图，标记所有存活对象       │               │
│  │                                          │               │
│  │  操作：                                   │               │
│  │  • 从 GC Roots 开始遍历                   │               │
│  │  • 标记所有可达对象                       │               │
│  │  • 使用写屏障处理并发变化                 │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段3: 最终标记（Final Mark）- STW ⚠️                      │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  目的：完成剩余标记工作                   │               │
│  │                                          │               │
│  │  操作：                                   │               │
│  │  • 处理并发标记期间的变化                 │               │
│  │  • 统计需要转移的 Region                  │               │
│  │  • 停顿时间：~5-10ms                     │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段4: 并发清理（Concurrent Cleanup）- 并发 ✅             │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  目的：清理完全垃圾的 Region              │               │
│  │                                          │               │
│  │  操作：                                   │               │
│  │  • 识别完全没有存活对象的 Region          │               │
│  │  • 将这些 Region 加入空闲列表             │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段5: 并发转移（Concurrent Evacuation）- 并发 ✅          │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  目的：将存活对象从收集集合转移到新 Region │               │
│  │                                          │               │
│  │  操作：                                   │               │
│  │  • 选择要转移的 Region（Collection Set）  │               │
│  │  • 分配新的空闲 Region                    │               │
│  │  • 复制存活对象到新 Region                │               │
│  │  • CAS 更新转发指针                       │               │
│  │                                          │               │
│  │  并发安全：                               │               │
│  │  • 使用转发指针保证并发安全               │               │
│  │  • 应用线程访问旧地址时，通过转发指针     │               │
│  │    找到新地址                             │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段6: 初始引用更新（Init Update Refs）- STW ⚠️            │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  目的：确保所有线程看到转发指针           │               │
│  │                                          │               │
│  │  操作：                                   │               │
│  │  • 确保所有线程到达安全点                 │               │
│  │  • 确保转发指针对所有线程可见             │               │
│  │  • 停顿时间：~1ms                        │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段7: 并发引用更新（Concurrent Update Refs）- 并发 ✅     │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  目的：更新所有指向旧位置的引用           │               │
│  │                                          │               │
│  │  操作：                                   │               │
│  │  • 扫描所有引用                           │               │
│  │  • 检查是否指向已转移的对象               │               │
│  │  • 如果是，更新为新地址                   │               │
│  │                                          │               │
│  │  使用 Brooks 转发指针：                   │               │
│  │  • 访问对象时，检查转发指针               │               │
│  │  • 如果转发指针 != 自己，跳转到新地址     │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段8: 最终引用更新（Final Update Refs）- STW ⚠️           │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  目的：完成引用更新                       │               │
│  │                                          │               │
│  │  操作：                                   │               │
│  │  • 处理剩余的引用更新                     │               │
│  │  • 停顿时间：~1ms                        │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                    ↓                                        │
│  阶段9: 并发清理（Concurrent Cleanup）- 并发 ✅             │
│  ┌─────────────────────────────────────────┐               │
│  │                                          │               │
│  │  目的：清理转移后的旧 Region              │               │
│  │                                          │               │
│  │  操作：                                   │               │
│  │  • 释放旧 Region 的内存                   │               │
│  │  • 更新 Region 状态                       │               │
│  │                                          │               │
│  └─────────────────────────────────────────┘               │
│                                                             │
│  总停顿时间：~5-15ms                                        │
│  • 初始标记：~1ms                                           │
│  • 最终标记：~5-10ms                                        │
│  • 初始引用更新：~1ms                                       │
│  • 最终引用更新：~1ms                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```


### 9.3 场景推荐

| 场景 | 推荐收集器 | 原因 |
|------|-----------|------|
| 桌面应用 | Serial | 堆小，停顿可接受 |
| Web 服务器 | G1 / ZGC | 低延迟，用户体验 |
| 大数据处理 | Parallel | 吞吐量优先 |
| 金融交易 | ZGC | 亚毫秒停顿 |
| 微服务 | G1 | 均衡表现 |
| 容器环境 | ZGC | 快速启动，低延迟 |
| 传统应用 | G1 | 开箱即用 |
| 性能测试 | Epsilon | 零开销 |

### 10.3 常见问题与解决方案

```
┌─────────────────────────────────────────────────────────────┐
│                  常见 GC 问题与解决方案                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  问题1: Full GC 频繁                                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  症状：                                              │   │
│  │  • Full GC 每分钟多次                                │   │
│  │  • 老年代使用率持续高位                               │   │
│  │                                                      │   │
│  │  原因：                                              │   │
│  │  • 老年代空间不足                                    │   │
│  │  • 内存泄漏                                          │   │
│  │  • 大对象分配过多                                    │   │
│  │                                                      │   │
│  │  解决：                                              │   │
│  │  • 增大堆大小：-Xmx4g                                │   │
│  │  • 增大老年代比例：-XX:NewRatio=3                    │   │
│  │  • 排查内存泄漏：jmap -dump:format=b,file=heap.bin  │   │
│  │  • 优化大对象：-XX:PretenureSizeThreshold=2m        │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  问题2: Young GC 频繁                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  症状：                                              │   │
│  │  • Young GC 每秒多次                                 │   │
│  │  • Eden 区快速填满                                   │   │
│  │                                                      │   │
│  │  原因：                                              │   │
│  │  • 新生代空间不足                                    │   │
│  │  • 对象分配速率过高                                  │   │
│  │                                                      │   │
│  │  解决：                                              │   │
│  │  • 增大新生代：-Xmn1g                                │   │
│  │  • 增大 Eden 比例：-XX:SurvivorRatio=8              │   │
│  │  • 优化代码，减少临时对象                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  问题3: 停顿时间过长                                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  症状：                                              │   │
│  │  • 单次 GC 停顿超过 1 秒                             │   │
│  │  • 用户体验差                                        │   │
│  │                                                      │   │
│  │  原因：                                              │   │
│  │  • 堆太大（传统收集器）                              │   │
│  │  • 存活对象太多                                      │   │
│  │  • 收集器不合适                                      │   │
│  │                                                      │   │
│  │  解决：                                              │   │
│  │  • 更换收集器：G1 / ZGC                              │   │
│  │  • 设置停顿目标：-XX:MaxGCPauseMillis=200           │   │
│  │  • 优化对象生命周期                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  问题4: 内存碎片                                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  症状：                                              │   │
│  │  • 老年代有空间但分配失败                            │   │
│  │  • Full GC 后碎片仍然存在                            │   │
│  │                                                      │   │
│  │  原因：                                              │   │
│  │  • 使用 CMS（标记-清除算法）                         │   │
│  │                                                      │   │
│  │  解决：                                              │   │
│  │  • 更换收集器：G1（标记-整理）                       │   │
│  │  • 启用压缩：-XX:+UseCMSCompactAtFullCollection     │   │
│  │  • 定期 Full GC 整理                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 附录：常用参数速查

### A.1 通用参数

```bash
# 堆大小
-Xms<size>                          # 初始堆大小
-Xmx<size>                          # 最大堆大小
-Xmn<size>                          # 新生代大小
-XX:NewSize=<size>                  # 新生代初始大小
-XX:MaxNewSize=<size>               # 新生代最大大小
-XX:NewRatio=<ratio>                # 老年代:新生代比例
-XX:SurvivorRatio=<ratio>           # Eden:S0:S1 比例

# 元空间
-XX:MetaspaceSize=<size>            # 元空间初始大小
-XX:MaxMetaspaceSize=<size>         # 元空间最大大小

# GC 日志
-Xlog:gc*:file=gc.log:time,uptime,level,tags
```

### A.2 收集器选择

```bash
-XX:+UseSerialGC                    # Serial GC
-XX:+UseParallelGC                  # Parallel GC
-XX:+UseConcMarkSweepGC             # CMS GC
-XX:+UseG1GC                        # G1 GC
-XX:+UseZGC                         # ZGC
-XX:+UseShenandoahGC                # Shenandoah GC
-XX:+UseEpsilonGC                   # Epsilon GC
```

### A.3 G1 参数

```bash
-XX:G1HeapRegionSize=<size>         # Region 大小
-XX:MaxGCPauseMillis=<ms>           # 最大停顿时间
-XX:InitiatingHeapOccupancyPercent=<pct> # 触发阈值
-XX:G1ReservePercent=<pct>          # 保留比例
-XX:G1HeapWastePercent=<pct>        # 浪费比例
```

### A.4 ZGC 参数

```bash
-XX:+ZGenerational                  # 分代 ZGC
-XX:SoftMaxHeapSize=<size>          # 软上限
-XX:ZCollectionInterval=<sec>       # 回收间隔
```

### A.5 监控工具

```bash
jstat -gcutil <pid> <interval>      # GC 统计
jmap -heap <pid>                    # 堆信息
jmap -dump:format=b,file=heap.bin <pid> # 堆转储
jstack <pid>                        # 线程转储
jcmd <pid> GC.run                   # 触发 GC
```
