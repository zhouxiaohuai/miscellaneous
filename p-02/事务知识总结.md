## 1. 事务基础概念

### ACID 四大特性

| 特性 | 说明 | 数据库实现 |
|------|------|-----------|
| **原子性 (Atomicity)** | 事务中的操作全部成功或全部失败，不可部分执行 | undo log（回滚日志） |
| **一致性 (Consistency)** | 事务前后数据从一个一致状态转为另一个一致状态 | 由原子性 + 隔离性共同保证 |
| **隔离性 (Isolation)** | 并发事务之间互不干扰 | 锁机制 + MVCC |
| **持久性 (Durability)** | 事务提交后数据永久保存 | redo log（重做日志） |

### Spring 事务管理核心接口

```
PlatformTransactionManager          # 事务管理器顶层接口
├── DataSourceTransactionManager    # JDBC / MyBatis 使用
├── JpaTransactionManager           # JPA / Hibernate 使用
└── HibernateTransactionManager     # Hibernate 原生使用
```

### 两种事务管理方式

| 方式 | 实现 | 特点 |
|------|------|------|
| **声明式** | `@Transactional` 注解 | 代码侵入小，基于 AOP 代理，推荐使用 |
| **编程式** | `TransactionTemplate` / `PlatformTransactionManager` | 精细控制，适用于复杂逻辑 |

## 2. 事务生命周期

### 声明式事务流程

```
方法调用 → AOP 代理拦截 → 获取事务 → 执行业务 → 提交/回滚 → 清理资源
```

**完整6个阶段：**

1. **解析阶段**：Spring 解析 `@Transactional` 注解，获取事务属性
2. **拦截阶段**：AOP 代理拦截方法调用，进入 `TransactionInterceptor`
3. **事务获取**：从 `PlatformTransactionManager` 获取事务（新建或加入已有事务）
4. **业务执行**：执行目标方法中的业务逻辑
5. **提交/回滚**：方法正常返回则提交，抛出异常则回滚（默认仅 `RuntimeException` / `Error`）
6. **清理阶段**：清理 `ThreadLocal` 中的事务信息，恢复挂起的事务

### 代码示例

**声明式事务 — 正常提交**

```java
@Transactional
public User declarativeTransactionDemo(User user) {
    User saved = userRepository.save(user);          // SQL 已执行，数据尚未持久化
    User found = userRepository.findByUsername(user.getUsername()).orElse(null);
    // 事务内查询可见，方法正常返回后 Spring 自动提交
    return saved;
}
```

**声明式事务 — 回滚**

```java
@Transactional
public User declarativeTransactionRollbackDemo(User user) {
    userRepository.save(user);
    throw new RuntimeException("模拟业务异常，触发事务回滚");  // RuntimeException 触发回滚
}
```

**编程式事务 — 手动控制**

```java
public User programmaticTransactionDemo(User user) {
    DefaultTransactionDefinition def = new DefaultTransactionDefinition();
    def.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRED);
    def.setTimeout(30);
    TransactionStatus status = transactionManager.getTransaction(def);  // 手动开启
    try {
        User saved = userRepository.save(user);
        transactionManager.commit(status);    // 手动提交
        return saved;
    } catch (Exception e) {
        transactionManager.rollback(status);  // 手动回滚
        throw e;
    }
}
```

### 关键要点

- `@Transactional` 默认只对 `RuntimeException` 和 `Error` 回滚，**不对 checked 异常回滚**
- 事务信息存储在 `ThreadLocal` 中，与当前线程绑定
- 编程式事务适用于需要在方法内部精确控制提交/回滚时机的场景

## 3. 7种传播行为

传播行为定义了当一个事务方法调用另一个事务方法时，事务如何传播。

### 总览表

| 传播行为 | 外部有事务 | 外部无事务 | 典型场景 |
|---------|-----------|-----------|---------|
| **REQUIRED**（默认） | 加入当前事务 | 新建事务 | 大多数业务方法 |
| **REQUIRES_NEW** | 挂起当前事务，新建独立事务 | 新建事务 | 审计日志、不受外层回滚影响 |
| **SUPPORTS** | 加入当前事务 | 以非事务方式执行 | 查询方法 |
| **NOT_SUPPORTED** | 挂起当前事务，以非事务方式执行 | 以非事务方式执行 | 不需要事务的操作 |
| **MANDATORY** | 加入当前事务 | 抛出异常 | 必须在事务中执行的核心操作 |
| **NEVER** | 抛出异常 | 以非事务方式执行 | 不允许在事务中执行的操作 |
| **NESTED** | 在当前事务中创建 Savepoint（嵌套事务） | 新建事务 | 部分失败不影响整体的场景 |

### 详细说明与代码示例

#### 3.1 REQUIRED（默认）

内部方法加入外部事务，**共用同一个事务**。内部异常会导致整体回滚。

```java
@Transactional  // 默认 REQUIRED
public void requiredOuter(User user1, User user2) {
    userRepository.save(user1);
    try {
        innerService.requiredInner(user2);  // 内部抛出异常
    } catch (Exception e) {
        // 即使捕获了内部异常，user1 也会被回滚！
        // 因为内外在同一个事务中，内部异常已标记事务为 rollback-only
    }
}
```

> **注意**：即使外部方法 catch 了内部异常，事务仍会回滚。因为内部方法已经将事务标记为 `rollback-only`，外部方法提交时仍会触发 `UnexpectedRollbackException`。

#### 3.2 REQUIRES_NEW

内部方法**始终新建独立事务**，外部事务被挂起。内部回滚不影响外部。

```java
@Transactional
public void requiresNewOuter(User user1, User user2) {
    userRepository.save(user1);
    try {
        innerService.requiresNewInner(user2);  // 独立事务，回滚不影响外部
    } catch (Exception e) {
        // user1 正常保存，user2 被回滚
    }
}
```

#### 3.3 SUPPORTS

有事务就加入，没事务就以非事务方式执行。

```java
@Transactional(propagation = Propagation.SUPPORTS)
public void supportsInner(User user) {
    userRepository.save(user);  // 有事务则加入，无事务则自动提交
}
```

#### 3.4 NOT_SUPPORTED

挂起当前事务，以非事务方式执行。适用于不需要事务保护的操作。

```java
@Transactional(propagation = Propagation.NOT_SUPPORTED)
public void notSupportedInner(User user) {
    userRepository.save(user);  // 始终以非事务方式执行
}
```

#### 3.5 MANDATORY

**必须**在事务中调用，否则抛出 `IllegalTransactionStateException`。

```java
@Transactional(propagation = Propagation.MANDATORY)
public void mandatoryInner(User user) {
    userRepository.save(user);  // 如果没有外部事务 → 抛异常
}
```

#### 3.6 NEVER

**不能**在事务中调用，否则抛出 `IllegalTransactionStateException`。

```java
@Transactional(propagation = Propagation.NEVER)
public void neverInner(User user) {
    userRepository.save(user);  // 如果有外部事务 → 抛异常
}
```

#### 3.7 NESTED

在当前事务中创建 Savepoint，内部回滚仅回到 Savepoint，外部事务可继续。

```java
@Transactional
public void nestedOuter(User user1, User user2) {
    userRepository.save(user1);
    try {
        innerService.nestedInner(user2);  // 回滚到 Savepoint
    } catch (Exception e) {
        // user1 仍可正常提交，user2 被回滚
    }
}
```

### REQUIRES_NEW vs NESTED 对比

| 对比项 | REQUIRES_NEW | NESTED |
|--------|-------------|--------|
| 事务关系 | 完全独立的新事务 | 外部事务的子事务（Savepoint） |
| 内部回滚 | 独立回滚，不影响外部 | 回滚到 Savepoint，外部可继续 |
| 外部回滚 | 不影响内部已提交的事务 | 内部一起回滚（因为是同一事务） |
| 资源消耗 | 需要额外数据库连接 | 共用同一连接 |
| 数据库支持 | 所有数据库 | 需要支持 Savepoint（InnoDB 支持） |

## 4. 4种隔离级别

隔离级别解决并发事务间的数据可见性问题。隔离级别越高，一致性越强，但并发性能越低。

### 总览表

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 性能 | MySQL 默认 |
|---------|------|-----------|------|------|-----------|
| **READ_UNCOMMITTED** | 可能 | 可能 | 可能 | 最高 | 否 |
| **READ_COMMITTED** | 避免 | 可能 | 可能 | 较高 | 否（PG 默认） |
| **REPEATABLE_READ** | 避免 | 避免 | 可能 | 中等 | **是（InnoDB 默认）** |
| **SERIALIZABLE** | 避免 | 避免 | 避免 | 最低 | 否 |

> InnoDB 在 REPEATABLE_READ 级别通过 MVCC + Next-Key Lock 在很大程度上避免了幻读。

### 并发问题详解

#### 4.1 脏读（READ_UNCOMMITTED）

事务A读到了事务B**未提交**的数据，事务B随后回滚，事务A读到的就是"脏数据"。

```
时间线:
事务A(READ_UNCOMMITTED)          事务B
├─ READ: value = 100
│                                ├─ UPDATE: value = 999 (未提交)
├─ READ: value = 999 ← 脏读！
│                                └─ ROLLBACK: value 恢复为 100
├─ READ: value = 100
```

```java
@Transactional(isolation = Isolation.READ_UNCOMMITTED)
public int dirtyRead_Write() {
    isolationTestRepository.updateValueByName("test_key", 999);  // 未提交
    throw new RuntimeException("故意回滚");  // 回滚后，脏读的数据无效
}
```

#### 4.2 不可重复读（READ_COMMITTED）

同一事务内，两次读取同一行数据结果不同（其他事务已提交的修改）。

```
时间线:
事务A(READ_COMMITTED)            事务B
├─ READ: value = 100
│                                ├─ UPDATE: value = 200
│                                └─ COMMIT
├─ READ: value = 200 ← 不可重复读！
```

```java
@Transactional(isolation = Isolation.READ_COMMITTED)
public int[] nonRepeatableRead_Read() {
    int firstRead = getCurrentValue();   // 100
    Thread.sleep(3000);
    int secondRead = getCurrentValue();  // 200（如果中间有其他事务修改并提交）
    return new int[]{firstRead, secondRead};
}
```

#### 4.3 幻读（REPEATABLE_READ）

同一事务内，两次范围查询的**行数**不同（其他事务已提交的新增/删除）。

```
时间线:
事务A(REPEATABLE_READ)           事务B
├─ SELECT COUNT(*) = 1
│                                ├─ INSERT new row
│                                └─ COMMIT
├─ SELECT COUNT(*) = 2 ← 幻读！
```

> InnoDB 通过 Next-Key Lock（行锁 + 间隙锁）在很大程度上避免幻读，但在特定场景下仍可能发生。

#### 4.4 SERIALIZABLE

最高隔离级别，完全串行化执行，避免所有并发问题，但性能最低。

```java
@Transactional(isolation = Isolation.SERIALIZABLE)
public int serializableDemo() {
    int currentValue = getCurrentValue();
    isolationTestRepository.updateValueByName("test_key", currentValue + 1);
    return currentValue + 1;
}
```

### 选择建议

| 业务场景 | 推荐隔离级别 | 原因 |
|---------|-------------|------|
| 报表查询 | READ_COMMITTED | 允许不可重复读，性能好 |
| 一般业务 | REPEATABLE_READ（默认） | 平衡一致性与性能 |
| 金融核心 | SERIALIZABLE | 绝对一致性，牺牲性能 |

## 5. 12种事务失效场景

这是本项目最重要的知识点，总结了 `@Transactional` 不生效的12种常见原因及解决方案。

### 速查表

| # | 场景 | 原因 | 解决方案 |
|---|------|------|---------|
| 1 | 方法非 public | AOP 只拦截 public 方法 | 改为 public |
| 2 | 自调用（this调用） | 未走代理，注解不生效 | 注入自身代理 / `AopContext.currentProxy()` / 移到其他 Service |
| 3 | 异常被 catch 未抛出 | Spring 看不到异常，正常提交 | catch 后重新 throw |
| 4 | 抛出 checked 异常 | 默认只回滚 RuntimeException/Error | `rollbackFor = Exception.class` |
| 5 | rollbackFor 配置错误 | 指定类型与实际异常不匹配 | `rollbackFor = Exception.class` |
| 6 | MyISAM 引擎 | MyISAM 不支持事务 | 使用 InnoDB |
| 7 | 未被 Spring 管理 | 手动 new 的对象不在容器中 | 使用 @Component/@Service + 注入 |
| 8 | 多线程调用 | ThreadLocal 无法跨线程传递事务 | 避免事务中开新线程 / 编程式事务 |
| 9 | propagation 设置错误 | 如 NOT_SUPPORTED 挂起事务 | 根据业务选择正确传播行为 |
| 10 | final/static 方法 | CGLIB 无法重写 final/static 方法 | 避免在 final/static 方法上用 @Transactional |
| 11 | 类未被代理 | 类不在 Spring 扫描路径或无注解 | @Component/@Service + 正确包路径 |
| 12 | BeanPostProcessor 顺序 | @Async 等切面先于事务切面处理 | @Order 控制顺序 / 避免混用 |

### 详细分析

#### 场景1：方法非 public

```java
@Transactional
protected void fail01_NotPublic(User user) {   // protected → 事务失效！
    userRepository.save(user);
    throw new RuntimeException("不会回滚");
}
```

**原理**：Spring AOP 基于 CGLIB 或 JDK 动态代理，只拦截 public 方法。

**解决**：将方法改为 `public`。

#### 场景2：自调用（this调用）

```java
public void outerMethod(User user) {
    this.innerMethod(user);  // this 调用未走代理！
}

@Transactional
public void innerMethod(User user) {
    userRepository.save(user);
    throw new RuntimeException("不会回滚");
}
```

**原理**：`this.innerMethod()` 直接调用目标对象的方法，绕过了 AOP 代理。

**解决方案**：
```java
// 方案1：注入自身代理
@Autowired
private TransactionFailService self;

public void outerMethod(User user) {
    self.innerMethod(user);  // 通过代理调用
}

// 方案2：AopContext.currentProxy()
public void outerMethod(User user) {
    ((TransactionFailService) AopContext.currentProxy()).innerMethod(user);
}

// 方案3：将事务方法移到另一个 Service 类
```

#### 场景3：异常被 catch 未抛出

```java
@Transactional
public void fail(User user) {
    userRepository.save(user);
    try {
        int i = 1 / 0;
    } catch (ArithmeticException e) {
        log.error("异常被捕获", e);  // Spring 看不到异常 → 正常提交！
    }
}

// 正确做法：
@Transactional
public void success(User user) {
    userRepository.save(user);
    try {
        int i = 1 / 0;
    } catch (ArithmeticException e) {
        throw e;  // 重新抛出 → 触发回滚
    }
}
```

#### 场景4：抛出 checked 异常

```java
@Transactional
public void fail(User user) throws IOException {
    userRepository.save(user);
    throw new IOException("checked 异常 → 不回滚！");  // IOException 不是 RuntimeException
}

// 正确做法：
@Transactional(rollbackFor = Exception.class)  // 指定回滚所有异常
public void success(User user) throws IOException {
    userRepository.save(user);
    throw new IOException("指定 rollbackFor 后会回滚");
}
```

**原理**：Spring 默认只对 `RuntimeException` 和 `Error` 回滚，`IOException` 等 checked 异常不回滚。

#### 场景5：rollbackFor 配置错误

```java
@Transactional(rollbackFor = IOException.class)
public void fail(User user) {
    userRepository.save(user);
    throw new RuntimeException("RuntimeException 不在 rollbackFor 范围内");
    // rollbackFor 指定 IOException，实际抛出 RuntimeException → 不回滚！
}
```

**解决**：统一使用 `@Transactional(rollbackFor = Exception.class)`。

#### 场景6：MyISAM 引擎

MyISAM 不支持事务，即使 `@Transactional` 正常，数据也不会回滚。

```sql
-- 检查表引擎
SHOW TABLE STATUS LIKE 't_user';
-- 修改为 InnoDB
ALTER TABLE t_user ENGINE = InnoDB;
```

#### 场景7：未被 Spring 管理

```java
// 错误：手动 new 的对象不在 Spring 容器中
NotSpringManaged obj = new NotSpringManaged();
obj.saveUser(userRepository, user);  // @Transactional 不生效

// 正确：通过 @Component 让 Spring 管理，通过 @Autowired 注入
```

#### 场景8：多线程调用

```java
@Transactional
public void fail(User user) {
    userRepository.save(user);
    CompletableFuture.runAsync(() -> {
        // 新线程中的数据库操作不受主线程事务控制
        // 事务信息存储在 ThreadLocal，新线程获取不到
        userRepository.save(otherUser);
    });
    throw new RuntimeException("主线程回滚，但子线程数据已提交");
}
```

**原理**：Spring 事务上下文存储在 `ThreadLocal` 中，线程间不共享。

**解决**：避免在事务方法中开启新线程操作数据库，或使用编程式事务在新线程中手动管理。

#### 场景9：propagation 设置错误

```java
@Transactional
public void outerMethod(User user) {
    userRepository.save(user);
    innerService.notSupportedMethod(otherUser);  // NOT_SUPPORTED 挂起事务
    throw new RuntimeException("外层回滚，但内层操作已非事务执行");
}
```

#### 场景10：final/static 方法

CGLIB 通过生成子类重写方法实现代理，`final` 方法不能被重写，`static` 方法属于类不属于实例，AOP 无法拦截。

```java
@Transactional
public final void fail(User user) {  // final → 事务失效
    userRepository.save(user);
}
```

#### 场景11：类未被代理

确保类被 `@Component`/`@Service`/`@Repository` 等注解标注，且在 Spring 的组件扫描路径下。

#### 场景12：BeanPostProcessor 顺序

当 `@Async` + `@Transactional` 同时使用时，如果 `@Async` 在 `@Transactional` 之前处理，异步方法在新线程执行，事务上下文丢失。

**解决**：使用 `@Order` 控制切面顺序，或避免 `@Async` 和 `@Transactional` 组合使用。

## 6. 编程式事务

当声明式事务无法满足需求时（如方法内需要精确控制提交/回滚时机），使用编程式事务。

### 方式1：TransactionTemplate（推荐）

```java
// 有返回值
public User templateWithReturn(User user) {
    return transactionTemplate.execute(status -> {
        User saved = userRepository.save(user);
        return saved;
    });
}

// 手动回滚
public void templateManualRollback(User user) {
    transactionTemplate.execute(status -> {
        userRepository.save(user);
        status.setRollbackOnly();  // 标记回滚
        return null;
    });
}
```

### 方式2：PlatformTransactionManager（底层API）

```java
public User managerComplete(User user) {
    DefaultTransactionDefinition def = new DefaultTransactionDefinition();
    def.setName("programmatic-tx");
    def.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRED);
    def.setTimeout(30);

    TransactionStatus status = transactionManager.getTransaction(def);
    try {
        User saved = userRepository.save(user);
        transactionManager.commit(status);
        return saved;
    } catch (Exception e) {
        transactionManager.rollback(status);
        throw e;
    }
}
```

### 对比

| 对比项 | TransactionTemplate | PlatformTransactionManager |
|--------|-------------------|--------------------------|
| 抽象层级 | 高（模板模式） | 低（手动管理） |
| 代码量 | 少 | 多 |
| 异常处理 | 自动回滚（RuntimeException） | 手动 rollback |
| 灵活性 | 中等 | 最高 |
| 推荐度 | 优先使用 | 极端复杂场景 |

## 7. 嵌套事务与 Savepoint

### NESTED 传播行为

NESTED 在当前事务中创建一个 Savepoint，内部回滚仅回到 Savepoint，外部事务可继续执行。

```java
@Transactional
public void nestedDemo(User outerUser, User innerUser) {
    userRepository.save(outerUser);       // 外层操作
    try {
        innerService.nestedInner(innerUser);  // 内层抛异常 → 回滚到 Savepoint
    } catch (Exception e) {
        // 外层事务继续，outerUser 正常提交
    }
}
```

### 手动 Savepoint

```java
public void manualSavepointDemo(User user1, User user2) {
    DefaultTransactionDefinition def = new DefaultTransactionDefinition();
    TransactionStatus status = transactionManager.getTransaction(def);
    try {
        userRepository.save(user1);                  // Step 1: 保存用户1
        Object savepoint = status.createSavepoint(); // Step 2: 创建 Savepoint
        try {
            userRepository.save(user2);              // Step 3: 保存用户2
            throw new RuntimeException("模拟异常");
        } catch (Exception e) {
            status.rollbackToSavepoint(savepoint);   // Step 4: 回滚到 Savepoint，用户2被撤销
        }
        transactionManager.commit(status);           // Step 5: 提交，用户1保存成功
    } catch (Exception e) {
        transactionManager.rollback(status);
    }
}
```

### REQUIRES_NEW vs NESTED vs Savepoint

| 对比项 | REQUIRES_NEW | NESTED (Savepoint) |
|--------|-------------|-------------------|
| 事务数 | 独立新事务 | 同一事务的子事务 |
| 连接数 | 需要新连接 | 共用同一连接 |
| 外层回滚 | 不影响已提交的内层 | 内层一起回滚 |
| 内层回滚 | 独立回滚 | 回滚到 Savepoint |
| 资源消耗 | 高（需额外连接） | 低 |

## 8. 只读事务

```java
@Transactional(readOnly = true)
public List<User> findAllReadOnly() {
    return userRepository.findAll();
}
```

### 优化原理

| 优化层面 | 说明 |
|---------|------|
| **Hibernate 层** | 跳过脏检查（dirty checking），减少内存开销 |
| **数据库层** | MySQL 可能使用 MVCC 快照读，不加排他锁 |
| **架构层** | 可路由到从库（读写分离），减轻主库压力 |

### 注意事项

- 只读事务中执行写操作，行为因数据库而异：MySQL InnoDB 可能允许写入但不会优化，某些数据库直接抛异常
- 适用于查询、报表、数据导出等场景

## 9. 超时事务

```java
@Transactional(timeout = 30)    // 30秒超时
public User normalTransaction(User user) {
    return userRepository.save(user);
}

@Transactional(timeout = 1)     // 1秒超时
public User timeoutTransaction(User user) {
    userRepository.save(user);
    Thread.sleep(3000);         // 超过1秒 → TransactionTimedOutException
    return user;
}
```

### 要点

- `timeout` 单位为秒，`-1` 表示使用数据库默认超时
- 超时计算从事务开始到方法执行完毕，包括连接获取、SQL 执行、锁等待
- 超时触发 `TransactionTimedOutException`，事务自动回滚
- 可与隔离级别组合使用：`@Transactional(timeout = 5, isolation = Isolation.READ_COMMITTED)`

## 10. 批量操作事务优化

### 三种策略对比

| 策略 | 实现方式 | 优点 | 缺点 | 适用场景 |
|------|---------|------|------|---------|
| **单事务批量** | `@Transactional` | 速度快，一次提交 | 全部回滚风险 | < 1000 条 |
| **分批提交** | 编程式事务，每 N 条提交 | 部分回滚，风险可控 | 代码稍复杂 | 1000~10000 条 |
| **逐条提交** | 编程式事务，每条单独 | 精确控制，单条失败不影响其他 | 性能最差 | 关键数据，不允许批量失败 |

### 分批提交代码示例

```java
public int batchInsertWithBatchCommit(List<User> users, int batchSize) {
    int totalProcessed = 0;
    for (int i = 0; i < users.size(); i += batchSize) {
        List<User> batch = users.subList(i, Math.min(i + batchSize, users.size()));
        DefaultTransactionDefinition def = new DefaultTransactionDefinition();
        TransactionStatus status = transactionManager.getTransaction(def);
        try {
            for (User user : batch) {
                userRepository.save(user);
            }
            transactionManager.commit(status);
            totalProcessed += batch.size();
        } catch (Exception e) {
            transactionManager.rollback(status);  // 仅回滚当前批次
            throw e;
        }
    }
    return totalProcessed;
}
```

### 分批大小建议

- 每批 100~500 条（需根据单条数据大小调整）
- 超过 10000 条建议分批提交 + 异步处理

## 13. 自定义事务框架 Tx

### 13.1 设计背景

Spring `@Transactional` 基于 AOP 代理，存在三大局限：

| 局限 | 原因 | 后果 |
|------|------|------|
| 自调用失效 | AOP 代理无法拦截同类内部调用 | 内部方法上的 @Transactional 不生效 |
| 方法级边界 | 代理只能在方法前后拦截 | 无法在方法内部精确控制事务范围 |
| 必须 public | CGLIB 代理无法拦截 private 方法 | private 方法上的 @Transactional 不生效 |

### 13.2 核心架构

```
┌─────────────────────────────────────────────────────────┐
│                    Tx（门面类）                           │
│         提供静态方法：readOnly() / writable()             │
├─────────────────────────────────────────────────────────┤
│                  TxBuilder（链式构建器）                   │
│      支持 .timeout() .propagation() .isolation()         │
├─────────────────────────────────────────────────────────┤
│                  TxContext（事务上下文）                   │
│         通过 ApplicationContextAware 获取容器             │
├─────────────────────────────────────────────────────────┤
│            PlatformTransactionManager                    │
│            Spring 底层事务管理器（已有）                    │
└─────────────────────────────────────────────────────────┘
```

### 13.3 使用方式

```java
// 只读事务
Tx.readOnly(() -> {
    List<User> users = userRepository.findAll();
});

// 读写事务
Tx.writable(() -> {
    user.setStatus(2);
    userRepository.save(user);
});

// 有返回值
User user = Tx.readOnly(() -> userRepository.findById(1L).orElse(null));

// 链式配置（动态超时、传播行为）
Tx.builder()
    .writable()
    .timeout(10)
    .propagation(Propagation.REQUIRES_NEW)
    .executeWithoutResult(() -> { ... });
```

### 13.4 方法内任意位置开启事务

```java
public void businessMethod() {
    // 1. 事务外：查询和校验
    User user = userRepository.findById(userId).orElseThrow();
    if (user.getBalance().compareTo(amount) < 0) {
        throw new RuntimeException("余额不足");
    }

    // 2. 远程调用（事务外）
    String result = remoteService.call();

    // 3. 【这里才开启事务】只包裹写操作
    Tx.writable(() -> {
        user.setBalance(user.getBalance().subtract(amount));
        userRepository.save(user);
        logRepository.save(new TxLog("DEDUCT", amount));
    });

    // 4. 事务外：异步通知
    asyncService.notify(user);
}
```

### 13.5 为什么 Tx 能解决 @Transactional 的问题？

| 问题 | @Transactional | Tx |
|------|----------------|-----|
| 自调用失效 | 依赖 AOP 代理拦截 | 静态方法调用，不依赖代理 |
| 方法级边界 | 代理只能在方法前后拦截 | 手动控制在哪个代码块开启事务 |
| 必须 public | CGLIB 代理限制 | 直接调用 API，无代理限制 |

**本质区别：** `@Transactional` 是 Spring 帮你调 `PlatformTransactionManager`，`Tx` 是你自己调 `PlatformTransactionManager`。

两者都是在当前线程内执行，底层都是同一个事务管理器，区别只是**谁来控制事务的开启和关闭时机**。

### 13.6 框架文件结构

```
src/main/java/com/example/transaction/framework/
├── Tx.java          # 门面类，提供静态方法
├── TxBuilder.java   # 链式构建器
├── TxContext.java   # 事务上下文，持有 TransactionManager
├── TxConfig.java    # 配置类
└── TxException.java # 自定义异常
```

### 13.7 事务框架对比

| 特性 | @Transactional | TransactionTemplate | Tx 框架 |
|------|----------------|--------------------|---------|
| 使用方式 | 注解 | 编程式 | 静态方法 |
| 事务边界 | 方法级 | 代码块级 | 代码块级 |
| 自调用 | 失效 | 正常 | 正常 |
| 访问限制 | 必须 public | 无限制 | 无限制 |
| 只读支持 | readOnly=true | 需手动设置 | readOnly() |
| 链式配置 | 不支持 | 不支持 | 支持 |
| 适用场景 | 简单场景 | 需要灵活控制 | 需要精确控制 |

## 14. ApplicationContextAware 机制

### 14.1 作用

**一句话：让 Spring Bean 能拿到 Spring 容器本身。**

### 14.2 工作流程

```
Spring 容器启动
    ↓
创建 Bean（如 TxContext）
    ↓
发现实现了 ApplicationContextAware
    ↓
自动调用 setApplicationContext(ctx)
    ↓
Bean 拿到容器引用，存到静态变量
    ↓
后续可通过该 Bean 获取任何其他 Bean
```

### 14.3 代码示例

```java
@Component
public class TxContext implements ApplicationContextAware {

    private static ApplicationContext applicationContext;

    @Override
    public void setApplicationContext(ApplicationContext ctx) {
        // Spring 自动调用，把容器传给你
        TxContext.applicationContext = ctx;
    }

    // 静态方法获取 Bean
    public static <T> T getBean(Class<T> clazz) {
        return applicationContext.getBean(clazz);
    }
}
```

### 14.4 为什么 Tx 框架需要它？

```java
// Tx.java 是工具类（不是 Spring Bean）
// 无法直接 @Autowired PlatformTransactionManager
// 解决方案：通过 TxContext 间接获取

public class Tx {
    static <T> T executeInternal(...) {
        PlatformTransactionManager txManager = TxContext.getTransactionManager();
        // ...
    }
}
```

| 类型 | 能否直接 @Autowired | 解决方案 |
|------|---------------------|----------|
| @Service / @Component | 能 | 直接注入 |
| 普通类（如 Tx） | 不能 | 通过 ApplicationContextAware 拿容器 |

### 14.5 常见用途

```java
// 1. 获取其他 Bean
UserService userService = applicationContext.getBean(UserService.class);

// 2. 获取配置信息
String appName = applicationContext.getEnvironment().getProperty("app.name");

// 3. 获取所有某类型的 Bean
Map<String, Handler> handlers = applicationContext.getBeansOfType(Handler.class);
```

**一句话总结：`ApplicationContextAware` 是 Spring 给你的"后门"，让你能在非 Spring 管理的类中访问容器。**

## 15. Runnable / Callable 与线程

### 15.1 常见误解

**误解：** `Runnable` 参数 = 新线程

**事实：** `Runnable` 只是一个函数式接口（一段代码的载体），本身和线程无关。

### 15.2 关键区别

```java
// 开新线程
new Thread(() -> {
    System.out.println("在新线程执行");
}).start();

// 当前线程直接调用
Runnable action = () -> {
    System.out.println("在当前线程执行");
};
action.run();  // ← 直接调用，不开新线程
```

### 15.3 判断标准

| 用法 | 是否新线程 |
|------|-----------|
| `new Thread(runnable).start()` | 是 |
| `executor.submit(runnable)` | 是 |
| `runnable.run()` | 否，当前线程 |
| `Tx.writable(runnable)` | 否，当前线程 |

### 15.4 Tx 框架的实现

```java
// Tx.java 核心代码
static <T> T executeInternal(Callable<T> action) {
    // 开启事务（当前线程）
    TransactionStatus status = txManager.getTransaction(def);

    try {
        // ★ 直接调用 action.call()，不是 new Thread(action).start()
        T result = action.call();  // ← 当前线程执行

        // 提交事务（当前线程）
        txManager.commit(status);
        return result;
    } catch (Exception e) {
        txManager.rollback(status);
        throw ...;
    }
}
```

**一句话总结：`Runnable` / `Callable` 只是一段代码的"包裹"，扔给 `Thread` 才开新线程，自己调 `run()` / `call()` 就是当前线程。**

### 16.1 事务框架选择建议

| 场景 | 推荐方式 |
|------|----------|
| 简单的 CRUD 方法 | @Transactional |

| 动态超时/传播行为 | Tx.builder()...execute() |

### 16.2 事务时间控制标准

```
正常事务：< 200ms
需要关注：200ms - 1s
WARNING：1s - 5s
CRITICAL：> 5s（必须优化）
```

### 16.3 大事务优化策略

1. **非DB操作外提** - RPC/文件操作放到事务外
2. **分片提交** - 大批量数据按每100-500条一个事务
3. **超时控制** - @Transactional(timeout=N) 或 Tx.builder().timeout(N)
4. **异步处理** - 日志/通知等非核心操作异步执行
5. **缩小事务范围** - 查询和校验在事务外，只把写操作放事务内

## 16. 大事务 + Tx 框架联动

### 16.1 核心思路

大事务的问题是**事务时间过长**，Tx 框架通过以下方式解决：

| 策略 | Tx 实现方式 | 效果 |
|------|------------|------|
| 分片提交 | 循环中每片一个 `Tx.writable()` | 每片独立事务，失败只回滚当前片 |
| 最小事务范围 | 查询在 `Tx.readOnly()`，写操作在 `Tx.writable()` | 事务只包裹写操作，时间最短 |
| 动态超时 | `Tx.builder().timeout(N)` | 根据数据量调整超时时间 |

### 16.2 分片提交

```java
// 批量导入 10000 条数据，每 500 条一个事务
for (int i = 0; i < allUsers.size(); i += chunkSize) {
    List<User> chunk = allUsers.subList(i, Math.min(i + chunkSize, allUsers.size()));

    // ★ 每片用 Tx.writable() 包裹，独立事务
    Tx.builder()
        .writable()
        .timeout(30)  // 每片独立超时
        .execute(() -> userRepository.saveAll(chunk));
}
```

**优势：**
- 每片事务时间短，连接占用少
- 某片失败只回滚当前片，不影响已提交的片
- Undo Log 及时回收

### 16.3 最小事务范围

```java
public void businessMethod(Long userId, BigDecimal amount) {
    // ===== 事务外：查询 =====
    User user = Tx.readOnly(() -> userRepository.findById(userId).orElseThrow());
    Account account = Tx.readOnly(() -> accountRepository.findByUserId(userId).orElseThrow());

    // ===== 事务外：校验 =====
    if (account.getBalance().compareTo(amount) < 0) {
        throw new RuntimeException("余额不足");
    }

    // ===== 事务外：远程调用 =====
    String rpcResult = remoteService.call();

    // ===== 事务内：只包含写操作 =====
    Tx.writable(() -> {
        accountRepository.updateBalance(account.getId(), amount.negate());
        logRepository.save(new TxLog("DEDUCT", amount));
    });

    // ===== 事务外：异步通知 =====
    asyncService.notify(user);
}
```

**效果：** 事务仅占总耗时的 3%（示例：总耗时 548ms，事务仅 19ms）

## 17. 嵌套事务 + Tx 框架联动

### 17.1 核心思路

嵌套事务通过 `Tx.builder().propagation()` 设置传播行为：

| 传播行为 | 用途 | 效果 |
|---------|------|------|
| `REQUIRES_NEW` | 独立事务 | 内层回滚不影响外层，外层回滚不影响内层 |
| `NESTED` | 嵌套事务 | 内层回滚到 Savepoint，外层可继续 |

### 17.2 REQUIRES_NEW：独立事务

```java
// 场景：批量处理用户，每个用户独立事务
for (Long userId : userIds) {
    try {
        // ★ 每个用户用 REQUIRES_NEW 独立事务
        Tx.builder()
            .writable()
            .propagation(Propagation.REQUIRES_NEW)  // 独立新事务
            .timeout(10)
            .execute(() -> {
                User user = userRepository.findById(userId).orElseThrow();
                user.setStatus(2);
                return userRepository.save(user);
            });
    } catch (Exception e) {
        // 某个用户失败，不影响其他用户
        log.error("用户 {} 处理失败", userId);
    }
}
```

**效果：** 每个用户独立事务，失败只回滚自己，不影响其他用户。

### 17.3 NESTED：嵌套事务（Savepoint）

```java
// 场景：主流程 + 子流程，子流程失败可回滚到 Savepoint

// 外层事务：创建用户
User outerUser = Tx.builder()
    .writable()
    .timeout(30)
    .execute(() -> userRepository.save(newUser));

// 内层事务：尝试更新（可能失败）
try {
    Tx.builder()
        .writable()
        .propagation(Propagation.NESTED)  // ★ 嵌套事务（Savepoint）
        .timeout(10)
        .executeWithoutResult(() -> {
            User user = userRepository.findById(outerUser.getId()).orElseThrow();
            user.setStatus(3);
            userRepository.save(user);
            throw new RuntimeException("模拟异常");
        });
} catch (Exception e) {
    // 内层回滚到 Savepoint，外层继续
    log.info("内层回滚，外层继续");
}

// 外层继续：记录日志
Tx.writable(() -> {
    logRepository.save(new TxLog("NESTED_DEMO", "SUCCESS"));
});
```

**效果：** 内层回滚到 Savepoint，外层事务正常提交。

### 17.4 组合使用：REQUIRES_NEW + NESTED

```java
// 场景：主流程 + 多个子流程

// 子流程1：REQUIRES_NEW - 独立事务（日志记录）
Tx.builder()
    .writable()
    .propagation(Propagation.REQUIRES_NEW)  // 独立事务
    .executeWithoutResult(() -> {
        logRepository.save(new TxLog("START", userId));
    });

// 子流程2：NESTED - 嵌套事务（业务处理，可回滚）
try {
    Tx.builder()
        .writable()
        .propagation(Propagation.NESTED)  // 嵌套事务
        .executeWithoutResult(() -> {
            // 业务处理，可能失败
        });
} catch (Exception e) {
    // 回滚到 Savepoint，外层继续
}

// 子流程3：REQUIRES_NEW - 独立事务（结束日志）
Tx.builder()
    .writable()
    .propagation(Propagation.REQUIRES_NEW)  // 独立事务
    .executeWithoutResult(() -> {
        logRepository.save(new TxLog("END", userId));
    });
```

**选择建议：**
- 日志记录 → `REQUIRES_NEW`（独立，不受业务回滚影响）
- 业务处理 → `NESTED`（可回滚到 Savepoint，外层继续）

### 17.5 Tx 嵌套调用

**问题：** 在 `Tx.executeWithoutResult()` 的 lambda 里面再调用 `Tx` 方法，是否合理？

**答案：合理，但需要注意传播行为的配合。**

```java
// 外层：NESTED
Tx.builder()
    .writable()
    .propagation(Propagation.NESTED)
    .executeWithoutResult(() -> {

        // 内层：默认 REQUIRED → 加入外层事务
        Tx.writable(() -> {
            // 和外层同一个事务
        });

        // 内层：REQUIRES_NEW → 独立新事务
        Tx.builder()
            .writable()
            .propagation(Propagation.REQUIRES_NEW)
            .executeWithoutResult(() -> {
                // 独立事务，不受外层影响
            });

        // 内层：NESTED → 嵌套 Savepoint
        Tx.builder()
            .writable()
            .propagation(Propagation.NESTED)
            .executeWithoutResult(() -> {
                // 新的 Savepoint
            });
    });
```

**传播行为配合规则：**

| 外层传播 | 内层传播 | 效果 |
|---------|---------|------|
| NESTED | REQUIRED（默认） | 内层加入外层事务，共用同一个 Savepoint |
| NESTED | REQUIRES_NEW | 内层独立新事务，挂起外层 |
| NESTED | NESTED | 内层创建新的 Savepoint |
| REQUIRES_NEW | 任意 | 内层加入内层事务（因为外层已是独立事务） |

| `GET /api/tx-advanced/nested/nested?userId=1` | NESTED 嵌套事务 |
